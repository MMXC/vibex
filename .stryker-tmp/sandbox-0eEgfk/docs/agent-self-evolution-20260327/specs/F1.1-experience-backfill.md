# Spec: F1.1 — 经验条目补录

## 功能概述
补充 T-010 ~ T-014 共 5 个历史任务的经验条目到 HEARTBEAT.md，建立可追溯的学习记录。

## 执行步骤

1. **扫描历史任务**
   - 读取 `/root/.openclaw/workspace-{agent}/HEARTBEAT.md` 中的 HB 记录
   - 提取 T-010 ~ T-014 的任务 ID、日期、任务类型

2. **识别经验**
   - 对每个任务，判断：成功经验是什么？失败教训是什么？
   - 经验类型: `J` (JTBD相关), `D` (DoD相关), `C` (协作相关), `T` (技术相关)

3. **写入 HEARTBEAT.md**
   - 找到「经验沉淀索引」表格
   - 按时间顺序追加条目，格式: `| E0XX | YYYY-MM-DD | <情境> | <经验> | <教训> |`

## 验收标准

```javascript
// test-f1.1.js
const fs = require('fs');
const content = fs.readFileSync('/root/.openclaw/workspace-pm/HEARTBEAT.md', 'utf8');
const entries = content.match(/^\| E\d+ \| \d{4}-\d{2}-\d{2} \|/gm) || [];
console.log(`经验条目数量: ${entries.length}`);
expect(entries.length).toBeGreaterThanOrEqual(5);
```

```bash
# 验证命令
grep -c "^| E" /root/.openclaw/workspace-pm/HEARTBEAT.md
# 期望输出: ≥ 5
```

## 经验条目参考（T-010~T-014 自检）

| 任务 | 经验 | 教训 |
|------|------|------|
| T-010 | 分析任务要先扫历史，节省重构时间 | 避免在未读上下文时盲目开始 |
| T-011 | 提案归档到 YYYYMMDD 目录，检索方便 | 不要散落在各 workspace 根目录 |
| T-012 | 飞书消息带话题 ID reply-to，确保线程连贯 | 直接发群组消息容易丢失上下文 |
| T-013 | cron 间隔要覆盖关键时段（凌晨/深夜） | 只配白天导致长任务漏检 |
| T-014 | 经验沉淀要写进收尾 checklist | 事后补录容易遗忘，应作为流程一部分 |

## DoD
- [ ] 至少 5 条经验条目存在
- [ ] 每条包含: 日期、情境、经验、教训
- [ ] 格式符合表格规范（7列）
- [ ] 对应任务 ID 可追溯
