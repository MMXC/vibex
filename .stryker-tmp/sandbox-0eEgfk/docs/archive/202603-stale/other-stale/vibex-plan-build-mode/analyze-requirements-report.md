# 需求分析报告：Plan/Build 双模式切换

**项目**: VibeX  
**分析师**: Analyst Agent  
**日期**: 2026-03-14  

---

## 1. 执行摘要

**结论**: 建议实现 Plan/Build 双模式切换，让用户在"规划优先"和"直接构建"之间选择，预计可提升复杂需求处理准确率 25%+。

**关键发现**:
- 竞品 bolt.new 已实现明确的 Plan/Build 模式分离
- 当前 VibeX 缺少规划阶段，直接进入生成，对复杂需求支持不足
- 用户反馈希望有更可控的生成流程

---

## 2. 问题定义

### 2.1 当前痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| 一键生成模式 | 用户输入需求后直接生成，缺少规划阶段 | 复杂需求准确率低 |
| 无法预览方案 | 生成前无法看到 AI 的理解 | 用户不确定 AI 是否理解正确 |
| 难以迭代 | 生成后修改成本高 | 用户体验差，放弃率高 |
| 缺乏控制感 | 全程 AI 主导，用户被动 | 专业用户不信任系统 |

### 2.2 问题边界

**影响范围**: 首页需求输入流程、确认页面、AI 生成流程

**不影响范围**: 
- 用户认证系统
- 项目管理系统
- 已生成项目的编辑功能

---

## 3. 现状分析

### 3.1 当前 VibeX 用户流程

```
用户输入需求 → AI 分析需求 → 生成限界上下文 → 生成领域模型 → 生成业务流程 → 确认 → 保存项目
```

**问题**: 
- 所有步骤自动执行，用户无法干预
- 复杂需求（如电商系统、ERP）容易生成错误
- 缺少"规划确认"环节

### 3.2 竞品对比

#### v0.dev (Vercel)

| 维度 | 设计 | 特点 |
|------|------|------|
| 入口 | 单一聊天框 "Ask v0 to build..." | 简单直接 |
| 模式 | 隐式规划 + 生成 | AI 自动规划，用户可干预 |
| 模板 | 丰富的模板库 | 降低新用户门槛 |
| 特性 | "Agentic by default" | AI 自动创建任务、连接数据库 |

**关键特性**: "Plan" 标签出现在 Agentic 功能中，表示 AI 会先规划再执行。

#### bolt.new (StackBlitz)

| 维度 | 设计 | 特点 |
|------|------|------|
| 入口 | 单一输入框 + 双按钮 | **Plan / Build now** |
| 模式 | **明确的 Plan/Build 分离** | 用户可选择模式 |
| 导入 | GitHub / Figma 导入 | 支持外部资产 |
| 特性 | "Plan" 先生成方案，"Build" 直接生成代码 | **核心差异点** |

**关键特性**: 
```
[Plan] 按钮 → 先生成方案，用户确认后再构建
[Build now] 按钮 → 直接生成代码，一键完成
```

### 3.3 模式差异对比

| 维度 | Plan 模式 | Build 模式 |
|------|-----------|------------|
| 目标 | 生成方案，获得确认 | 直接生成可用代码 |
| 输出 | 需求理解、技术方案、功能清单 | 完整可运行项目 |
| 用户参与 | 高（需确认每个环节） | 低（自动完成） |
| 适用场景 | 复杂项目、企业应用 | 简单页面、快速原型 |
| 时间成本 | 较长（多轮确认） | 较短（一键完成） |
| 准确率 | 高（逐步确认） | 中等（可能需多次迭代） |

## 4. 方案对比

### 方案 A: 简化版 Plan/Build

**设计**: 在首页输入框旁添加模式切换开关

```
[需求输入框]                    [Plan 📋] [Build 🚀]
```

**Plan 模式流程**:
1. 用户输入需求
2. AI 生成理解摘要 + 功能清单
3. 用户确认/修改
4. 生成限界上下文
5. 用户确认
6. 生成领域模型
7. 最终生成项目

**Build 模式流程**:
1. 用户输入需求
2. 自动生成完整项目（当前流程）

**优点**: 实现简单，UI 变化小  
**缺点**: Plan 模式功能有限，用户可能不理解差异

---

### 方案 B: 完整版 Plan/Build

**设计**: 双入口 + 独立工作区

**Plan 模式** (规划工作区):
- 需求理解面板
- 功能分解树
- 技术方案预览
- WBS（工作分解结构）
- 用户确认检查点

**Build 模式** (快速生成):
- 保持当前一键生成流程
- 添加"基于 Plan 构建"选项

**优点**: 功能完整，用户体验好  
**缺点**: 开发工作量大，需要新组件

---

### 方案对比表

| 维度 | 方案 A (简化版) | 方案 B (完整版) |
|------|----------------|----------------|
| 开发工作量 | 3 人日 | 10 人日 |
| 用户学习成本 | 低 | 中 |
| 功能完整度 | 60% | 100% |
| 预期准确率提升 | 15% | 25%+ |
| 风险 | 低 | 中 |

**推荐**: 先实施方案 A（MVP），验证效果后再迭代至方案 B。

## 6. 用户场景

### 场景 1: 新用户快速体验

**用户**: 小明，第一次使用 VibeX  
**需求**: "帮我做一个个人博客"  
**选择**: Build 模式  
**结果**: 快速生成可运行的博客，体验成功后注册账号

### 场景 2: 企业用户规划项目

**用户**: 产品经理小李  
**需求**: "需要开发一个内部 CRM 系统，包含客户管理、销售漏斗、报表功能"  
**选择**: Plan 模式  
**流程**: 
1. AI 生成需求理解
2. 小李确认并补充细节
3. AI 生成技术方案
4. 小李确认技术选型
5. 开始构建

**结果**: 生成符合预期的 CRM 系统，无需多次迭代

### 场景 3: 复杂需求修正

**用户**: 技术负责人小王  
**需求**: "电商后台，支持多商户、库存管理、订单处理"  
**选择**: Plan 模式  
**流程**: 
1. AI 生成需求理解
2. 小王发现 AI 漏掉"多商户"理解
3. 修改需求描述
4. 重新生成理解
5. 确认后开始构建

**结果**: 避免了一次错误的生成，节省时间
