## 3. 技术评估

### 3.1 为什么"ESM 问题"被误报

1. **ESM vs CommonJS 混淆**：`ts-jest` 将 TypeScript 编译为 CommonJS，开发者可能误以为需要 ESM
2. **console.error 干扰**：错误处理测试中的 `console.error` 被 CI 日志误认为失败
3. **缺少 `--verbose` 过滤**：未区分"测试框架错误"和"应用代码日志"

### 3.2 当前配置的优劣

| 优点 | 缺点 |
|------|------|
| ✅ 兼容性好 | ❌ 无法使用原生 ESM 语法 |
| ✅ Jest 生态完善 | ⚠️ 与前端 Vite/ESM 生态不一致 |
| ✅ 测试通过，覆盖率达标 | |
| ✅ 配置简单 | |

### 3.3 如果要迁移到 ESM

如果要使用 `"type": "module"` + ESM 模式：

```javascript
// jest.config.ts (ESM)
export default {
  preset: 'ts-jest/presets/default-esm',
  testEnvironment: 'node',
  extensionsToTreatAsEsm: ['.ts'],
  // 需要额外的 transforms 和 moduleNameMapper
};
```

**风险**：中等，涉及大量配置变更和潜在破坏
