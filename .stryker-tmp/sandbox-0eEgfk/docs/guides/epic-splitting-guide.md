# Epic 拆分指南

> 如何将功能需求拆分为可执行的 Epic | 版本: 1.0 | 更新日期: 2026-03-17

---

## 1. Epic 粒度原则

### 目标工时: 1-2 小时

Epic 是介于 Feature 和 Story 之间的单位，应该:
- ✅ 可在 1-2 个工作小时内完成
- ✅ 有明确的交付物
- ✅ 可独立测试
- ✅ 有清晰的验收标准

### 粒度过大的信号

| 信号 | 问题 | 解决方案 |
|------|------|----------|
| 预估 > 4h | 粒度过大 | 拆分为多个 Epic |
| 涉及多个模块 | 职责不清 | 按模块拆分 |
| 依赖其他 Epic | 阻塞风险 | 调整拆分逻辑 |

---

## 2. 拆分方法

### 2.1 按功能模块拆分

```
原始: F1: 用户管理系统
├── Epic 1.1: 用户注册功能 (1h)
├── Epic 1.2: 用户登录功能 (1h)
├── Epic 1.3: 用户登出功能 (0.5h)
└── Epic 1.4: 密码找回功能 (2h)
```

### 2.2 按技术层次拆分

```
原始: F2: 首页性能优化
├── Epic 2.1: 减少首屏请求数 (1.5h)
├── Epic 2.2: 图片懒加载实现 (1h)
└── Epic 2.3: 缓存策略优化 (1h)
```

### 2.3 按用户流程拆分

```
原始: F3: 订单流程
├── Epic 3.1: 加入购物车 (1h)
├── Epic 3.2: 结算页面 (1.5h)
├── Epic 3.3: 支付流程 (2h)
└── Epic 3.4: 订单确认页 (1h)
```

---

## 3. Story 拆分

每个 Epic 应进一步拆分为 Stories:

### Story 格式

```
- S{n}.{m}: [具体任务描述]
```

### 示例

```
Epic 1.1: 用户注册功能
Stories:
- S1.1.1: 创建注册表单组件
- S1.1.2: 实现表单验证逻辑
- S1.1.3: 调用注册 API
- S1.1.4: 注册成功跳转
```

---

## 4. 优先级定义

| 优先级 | 定义 | 交付时间 |
|--------|------|----------|
| P0 | 核心功能，必需 | 本周 |
| P1 | 重要功能 | 下周 |
| P2 | 增强功能 | 下月 |
| P3 | 锦上添花 | 后续迭代 |

---

## 5. 验收标准关联

每个 Epic 必须关联对应的验收标准:

```
Epic 1.1: 用户注册功能
验收标准:
- AC1.1.1: 邮箱格式校验通过
- AC1.1.2: 密码强度要求满足
- AC1.1.3: 注册成功后跳转首页
```

---

## 6. 自检清单

- [ ] Epic 预估工时在 1-2h 范围内
- [ ] 每个 Epic 有明确的交付物
- [ ] Epic 之间无强依赖
- [ ] 每个 Epic 关联验收标准
- [ ] 优先级分配合理
