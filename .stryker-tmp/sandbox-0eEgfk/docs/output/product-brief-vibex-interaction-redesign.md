# Product Brief: VibeX 交互重构

**版本**: v1.0  
**日期**: 2026-03-10  
**作者**: Analyst  
**状态**: Draft

---

## 1. Problem Statement (核心痛点)

### 当前问题
| 痛点 | 影响 | 严重度 |
|------|------|--------|
| 交互割裂，项目上下文丢失 | 用户不知道领域模型/流程/原型属于哪个项目 | 🔴 高 |
| Dashboard 只能看项目列表 | 无法快速了解项目内容，需要多次跳转 | 🟡 中 |
| 新用户无法快速体验能力 | 缺少 "Wow 时刻"，转化率低 | 🔴 高 |
| 产出物无法直接交付团队 | 需要额外整理文档和原型 | 🟡 中 |

### 用户反馈原文
> "我只能看 dashboard 项目列表，然后这个页面的领域模型、原型、流程都不知道是哪个项目的，超级乱"

---

## 2. Proposed Solution (解决方案)

### 核心设计理念
**协作式 DDD 建模，让用户深度参与设计过程**

**与竞品的核心差异:**

| 维度 | 竞品（Uxbot/v0/lovable） | VibeX |
|------|--------------------------|-------|
| 生成方式 | 黑盒一键生成 | 分步协作生成 |
| 用户参与 | 被动接受结果 | 主动调整每一步 |
| 输出质量 | 碰运气 | 精确可控 |
| 学习价值 | 无 | 掌握 DDD 思维 |

**用户旅程:**

```
┌─────────────────────────────────────────────────────────────┐
│                 VibeX 协作式设计流程                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 需求澄清                                            │
│  用户输入模糊需求 → AI 提问澄清 → 需求摘要确认                 │
│      ↓                                                       │
│  Step 2: 领域模型（DDD）                                      │
│  AI 生成初始实体 → 用户增删实体/调整关系 → 确认领域模型        │
│      ↓                                                       │
│  Step 3: 业务流程                                            │
│  基于模型生成流程图 → 用户增删节点/调整分支 → 确认业务流程     │
│      ↓                                                       │
│  Step 4: UI 页面结构                                         │
│  基于流程生成页面 → 用户增删组件/调整布局 → 确认 UI 结构       │
│      ↓                                                       │
│  Step 5: 原型生成                                            │
│  生成可交互原型 → 用户微调细节 → 创建项目                      │
│      ↓                                                       │
│  Step 6: 导出交付                                            │
│  PRD 文档 + 静态原型链接                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**每一步的核心价值:**
- **领域模型**: 让用户理解系统有哪些核心概念，建立共识
- **业务流程**: 让用户梳理操作路径，发现遗漏场景
- **UI 结构**: 让用户参与界面设计，精确表达需求
- **原型**: 最终呈现，可直接交付团队

### 关键改动

| 改动点 | 现状 | 目标 |
|--------|------|------|
| 首页 | Dashboard 项目列表 | 对话入口 + 最近项目 |
| 项目结构 | 孤立的数据碎片 | 项目容器（需求/模型/流程/原型） |
| 对话模式 | 随处对话 | 项目内对话（针对具体内容） |
| 导出能力 | 无 | PRD + 静态原型链接 |

### 改版范围
- ✅ 交互体验重构
- ✅ 部分功能重构（项目容器化）
- ⏸️ 后端保持不变（模型可配置化）

## 3. Success Metrics (成功指标)

### 北极星指标
**新用户 7 日激活率** = 完成首次需求 → 原型生成 → 导出的用户比例

### 过程指标
| 指标 | 定义 | 目标 |
|------|------|------|
| 首次生成转化率 | 输入需求 → 等待生成 → 查看结果 的用户比例 | ≥ 60% |
| 项目创建率 | 查看原型 → 创建项目 的用户比例 | ≥ 40% |
| 导出率 | 创建项目 → 导出交付 的用户比例 | ≥ 20% |
| 用户满意度 | NPS 或评分 | ≥ 4.0/5.0 |

### 质量指标
| 指标 | 定义 | 目标 |
|------|------|------|
| 生成时间 | 需求输入 → 原型可用 | ≤ 5 分钟 |
| 原型可用性 | 用户认为原型"可直接展示给团队"的比例 | ≥ 70% |

## 4. Core User Stories (核心用户故事)

### Epic 1: 需求澄清阶段

**US-1.1: 需求录入与澄清**
> 作为新用户，我希望能通过对话逐步澄清模糊需求，以便系统准确理解我的意图。

**验收条件:**
- 输入框醒目，有引导文案和示例需求
- 系统主动提问澄清关键信息（每次 ≤3 个问题）
- 澄清过程有进度指示（如：已收集 3/5 关键信息）
- 用户可随时跳过或补充信息
- 澄清完成后生成需求摘要，用户确认后进入下一步

**澄清流程设计:**
```
用户输入模糊需求
    ↓
系统分析缺失信息
    ↓
系统提问（优先级排序，每次 ≤3 个问题）
    ↓
用户回答 / 跳过
    ↓
循环直到信息足够 或 用户确认"继续"
    ↓
生成需求摘要 → 用户确认
```

---

### Epic 2: 领域模型阶段（DDD 核心差异化）

**US-2.1: 生成初始领域模型**
> 作为用户，我希望看到系统根据需求生成的领域模型，以便了解系统核心概念。

**验收条件:**
- 展示实体列表（实体名称 + 核心属性）
- 展示实体关系图（连线表示关系类型）
- 每个实体有简要说明
- 支持缩放、拖拽查看

**US-2.2: 实体增删调整**
> 作为用户，我希望能增删实体和调整关系，以便领域模型准确反映业务逻辑。

**验收条件:**
- **对话操作**: "添加一个订单实体"、"删除商品分类"
- **代码操作**: 可编辑实体定义 JSON
- 支持添加/删除实体
- 支持修改实体属性
- 支持调整实体关系（一对一、一对多、多对多）
- 修改后实时更新图表

**US-2.3: 确认领域模型**
> 作为用户，我希望能确认领域模型后再进入下一步，以便建立团队共识。

**验收条件:**
- 展示完整的领域模型摘要
- 用户可返回修改
- 确认后进入流程设计阶段

---

### Epic 3: 业务流程阶段

**US-3.1: 生成初始业务流程**
> 作为用户，我希望看到基于领域模型生成的业务流程，以便梳理操作路径。

**验收条件:**
- 自动从领域模型推导核心流程
- 展示流程图（节点 + 连线）
- 标注关键决策点
- 支持多个流程（如：下单流程、退款流程）

**US-3.2: 流程节点增删**
> 作为用户，我希望能增删流程节点和调整分支，以便完善业务流程。

**验收条件:**
- **对话操作**: "在支付后添加库存检查"、"删除取消订单分支"
- **代码操作**: 可编辑流程定义
- 支持添加/删除节点
- 支持添加条件分支
- 支持调整节点顺序
- 修改后实时更新

**US-3.3: 确认业务流程**
> 作为用户，我希望能确认业务流程后再进入下一步，以便确保流程完整。

**验收条件:**
- 展示完整的流程摘要
- 标注可能遗漏的场景
- 用户可返回修改
- 确认后进入 UI 设计阶段

---

### Epic 4: UI 页面结构阶段

**US-4.1: 生成初始页面结构**
> 作为用户，我希望看到基于流程生成的页面列表，以便了解系统界面组成。

**验收条件:**
- 自动从流程推导页面
- 展示页面树形结构
- 每个页面标注来源（对应哪个流程节点）
- 支持页面预览

**US-4.2: UI 节点增删**
> 作为用户，我希望能增删页面组件，以便精确设计界面结构。

**验收条件:**
- **对话操作**: "在登录页添加记住我复选框"、"删除商品详情页的相关推荐"
- **代码操作**: 可编辑组件配置
- 支持添加/删除页面
- 支持添加/删除页面内组件
- 支持调整组件布局
- 修改后实时预览

**US-4.3: 确认 UI 结构**
> 作为用户，我希望能确认 UI 结构后再生成原型，以便控制最终效果。

**验收条件:**
- 展示完整的页面列表
- 每个页面可预览结构
- 用户可返回修改
- 确认后进入原型生成阶段

---

### Epic 5: 原型生成与导出

**US-5.1: 生成可交互原型**
> 作为用户，我希望看到最终的可交互原型，以便评估和分享。

**验收条件:**
- 基于确认的 UI 结构生成原型
- 原型可交互（点击跳转、表单输入）
- 可微调细节（颜色、字号、间距）
- 生成时间 ≤ 5 分钟

**US-5.2: 创建项目**
> 作为用户，我希望能将所有产出保存为项目，以便后续继续完善。

**验收条件:**
- 一键创建项目
- 项目包含：需求摘要、领域模型、业务流程、UI 结构、原型
- 项目可继续编辑

**US-5.3: 导出 PRD 文档**
> 作为用户，我希望一键导出 PRD 文档，以便交付给开发团队。

**验收条件:**
- PRD 包含：需求背景 + 领域模型 + 业务流程 + 交互说明 + 数据契约
- 支持 Markdown / PDF 格式
- 文档结构与设计阶段对应

**US-5.4: 导出静态原型**
> 作为用户，希望能生成可分享的静态原型链接，以便团队评审。

**验收条件:**
- 原型可独立访问（无需登录）
- 链接可设置有效期
- 支持密码保护（可选）

## 5. 竞品参考

### Uxbot.cn
- **核心体验**: 描述产品想法 → 高保真界面 + 交互原型 → 一键导出代码
- **亮点**: 云端模拟运行、标准化代码输出
- **学习点**: 简洁的对话入口、即时可视化反馈

### V0.dev
- **核心体验**: 描述 UI → 生成 React 代码 → 实时预览
- **亮点**: 极简对话、代码可直接使用
- **学习点**: 生成速度、代码质量

### Lovable.dev
- **核心体验**: 描述想法 → 完整应用 → 部署上线
- **亮点**: 从想法到上线全流程
- **学习点**: 端到端体验

## 6. 技术约束

| 约束 | 说明 |
|------|------|
| 后端不变 | 保持现有 API 和生成逻辑 |
| 模型可配置 | 支持切换不同 AI 模型 |
| 静态部署 | 原型部署到 Cloudflare Pages |
| 数据格式 | 保持现有 D1 数据库结构 |

## 7. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 生成质量不稳定 | 中 | 高 | 引入评分机制，低分重新生成 |
| 用户期望过高 | 中 | 中 | 明确"原型"定位，引导合理预期 |
| 迁移成本 | 低 | 中 | 保留旧版入口，渐进式迁移 |
| 模型切换复杂 | 低 | 低 | 统一抽象层，配置化管理 |
