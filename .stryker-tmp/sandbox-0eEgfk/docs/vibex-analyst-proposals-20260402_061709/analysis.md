## 1. 业务场景分析

### 1.1 VibeX 目标用户

| 用户类型 | 核心需求 | 关键痛点 |
|----------|----------|----------|
| 产品经理 | 快速将业务需求转化为可验证的原型 | 无结构化流程，输出物难传承 |
| 技术负责人 | 协作式 DDD 建模，跨团队对齐限界上下文 | 缺乏可视化，沟通成本高 |
| 前端开发者 | 从领域模型直接生成组件树，减少返工 | 设计-实现断层，手动映射繁琐 |

**目标用户画像**: 5-50 人规模的 SaaS 产品团队，核心场景是「需求起点 → 可交互原型」的快速迭代。

### 1.2 核心价值

VibeX 的核心竞争力是**协作式 DDD 建模 + AI 驱动的原型生成**，核心价值链：

```
用户输入需求 → AI 识别限界上下文 → 生成领域模型 → 生成业务流程 → 生成组件树
```

当前系统已跑通核心流程，但「最后一公里」存在三类系统性风险：状态模型分裂、确认机制碎片化、代码库膨胀。这三类风险如果不治理，会在团队规模扩大时指数级放大维护成本。

### 1.3 当前系统性风险等级评估

| 风险类别 | 当前等级 | 恶化速度 | 修复成本 |
|----------|----------|----------|----------|
| 三树选择模型不统一 | 🔴 高 | 快（每次新功能叠加） | 中（4-6h） |
| canvasStore 膨胀 | 🔴 高 | 快（持续堆叠） | 高（8-12h） |
| Canvas 布局规范缺失 | 🟡 中 | 中 | 中（6-8h） |
| 交互反馈碎片化 | 🟡 中 | 中 | 中（4-6h） |
| 测试覆盖不足 | 🟡 中 | 慢（显式积累） | 中（8-10h） |

## 2. 核心 JTBD（Jobs To Be Done）

### JTBD-1: 保持用户操作一致性
> **用户目标**: 在三棵树之间切换时，不需要重新学习「怎么选择、怎么确认」  
> **当前摩擦**: 三棵树 checkbox 位置不同、确认状态视觉反馈各异  
> **成功标准**: 任意用户首次使用任意一棵树，5分钟内无需帮助即可完成「选择→确认」全流程

### JTBD-2: 快速定位和修复 bug
> **用户目标**: 发现 Canvas 异常时，能快速定位是哪棵树的哪个状态出了问题  
> **当前摩擦**: canvasStore 1433 行，所有状态混在一起，状态更新链路不透明  
> **成功标准**: 任意状态相关 bug，从定位到修复不超过 1 小时（当前可能需要半天）

### JTBD-3: 高效扩展新功能
> **用户目标**: 团队新成员能快速上手 Canvas 开发，不需要全局上下文  
> **当前摩擦**: 修改一棵树的状态逻辑，可能意外影响另一棵树的展示  
> **成功标准**: 新增一个树节点类型（如 FlowNode 子类），代码改动量 < 50 行，且无回归

### JTBD-4: 降低 UI 变更的回归风险
> **用户目标**: 修改 Canvas 样式时，有明确的影响范围评估工具  
> **当前摩擦**: 改 CSS 后不确定哪些组件受影响，需要人工截图对比  
> **成功标准**: UI 变更后 5 分钟内完成影响范围扫描，并生成截图 diff

### JTBD-5: 团队共享统一的建模规范
> **用户目标**: 所有团队成员使用相同的验收标准编写 Story，不需要返工对齐  
> **当前摩擦**: 验收标准不具体（"用户可以..."），导致实现结果与预期不符  
> **成功标准**: 每个 Story 的验收标准可直接转化为 Playwright `expect()` 断言

## 3. 技术方案选项

### 方案 A: 渐进式重构（推荐）

**核心理念**: 不破坏现有功能，按优先级逐步拆分和规范化。

**实施路线图**:

| 阶段 | 内容 | 优先级 | 工时 | 产出 |
|------|------|--------|------|------|
| Phase 1 | 三树选择模型统一（NodeState 枚举 + checkbox 规范） | P0 | 4-6h | 统一的节点状态机定义 |
| Phase 2 | canvasStore 按领域拆分（contextStore → flowStore → componentStore → uiStore） | P0 | 8-12h | 4 个独立 store，canvasStore 降为代理层 |
| Phase 3 | Canvas 布局规范落地（sticky 工具栏 + 抽屉协议 + 面板动画） | P1 | 6-8h | 布局规范文档 + 实现 |
| Phase 4 | 交互反馈标准化（Feedback Token + toast 替换 confirm） | P1 | 4-6h | Feedback Token 规范 + 全局替换 |
| Phase 5 | 按用户旅程拆 E2E（3 个 spec: create-context / generate-flow / multi-select） | P2 | 8-10h | 3 个可运行的 E2E 测试 |
| Phase 6 | PRD 模板落地（GIVEN/WHEN/THEN 强制模板） | P2 | 3-4h | Story 模板 + lint 规则 |
| Phase 7 | 设计系统一致性审计（emoji → SVG + Spacing Token） | P3 | 6-8h | DESIGN.md 完整版 |

**总工时**: 39-54h（约 5-7 个工作日）

**优点**:
- 风险可控，每个 Phase 独立验收
- 不阻断当前开发节奏
- 可与日常迭代并行

**缺点**:
- 总工时较长（约 5-7 天）
- 需要 discipline 坚持走完所有 Phase

---

### 方案 B: 大爆炸重构（不推荐）

**核心理念**: 一次性完成所有规范化，重构期间冻结功能开发。

**实施路线图**:

| 阶段 | 内容 | 工时 |
|------|------|------|
| Week 1 | canvasStore 完全重写（并行开发），三树同步适配 | 20-25h |
| Week 2 | Canvas 布局重构 + 交互反馈标准化 + 组件设计系统 | 20-25h |
| Week 3 | 测试覆盖 + PRD 规范 + CI 集成 | 15-20h |

**总工时**: 55-70h（约 7-9 个工作日）

**优点**:
- 重构完成后代码库干净
- 可设计统一的架构视图

**缺点**:
- 冻结开发 2-3 周，业务损失大
- 重构范围太大，merge 冲突风险高
- 一旦出问题，回滚成本极高

**结论**: 方案 A 明显优于方案 B。渐进式重构可以将风险分散到每个 Phase，且每个 Phase 完成后都有可验证的产出。

## 7. 实施建议

### 建议 1: 合并三方提案，减少重复工作

Analyst / PM / Dev 三方提案存在大量重叠，建议合并为统一实施计划：

```
Sprint 0 (前置): D-001 + D-002 (TS清理 + Jest稳定) — 0.5天
Sprint 1: VS-1 (三树统一) + VS-4 (交互反馈) — 1天
Sprint 2: VS-2 (canvasStore拆分) — 1.5天 ← 最大风险点，需单独测试
Sprint 3: VS-3 (Canvas信息架构) + PM-P2 (UI变更checklist) — 1天
Sprint 4: VS-5 (E2E覆盖) + VS-6 (PRD模板) — 1天
Sprint 5: VS-7 (设计系统审计) + PM-P5 (Sprint回顾自动化) — 1天
```

**总计**: 约 6 个 Sprint（视团队节奏调整），约 6 人天的直接开发工作量。

### 建议 2: 每个 Phase 设置独立的可测试产出

不要等所有改动完成后才验收，每个 Phase 完成后立即验收对应的 VS 标准。这样可以：
- 早期发现回归问题
- 给团队提供可见的进展
- 减少最终集成时的风险

### 建议 3: 将 TS 错误清理设为 P0 前置

Dev D-001（9个预存 TS 错误）直接影响 CI 门禁的可靠性。在开始任何重构前，必须先修复这些错误，否则无法判断新的 CI 失败是重构引入还是历史遗留。
