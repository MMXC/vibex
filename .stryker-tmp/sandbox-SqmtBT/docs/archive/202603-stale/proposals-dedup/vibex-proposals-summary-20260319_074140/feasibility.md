### 2.1 A-001: API 服务层按领域拆分

#### 可行性分析

**技术可行性**: ✅ **可行**

| 评估维度 | 结论 | 依据 |
|----------|------|------|
| 技术复杂度 | 中等 | 纯 TypeScript 模块化拆分，无需引入新框架 |
| 团队能力 | 具备 | Dev Agent 具备 Next.js 和 TypeScript 能力 |
| 依赖外部 | 无 | 仅使用内置 Node.js 模块和现有依赖 |
| 迁移风险 | 可控 | 兼容层策略确保现有调用不受影响 |

**实现路径**:
```
Phase 1: auth.ts 拆分 (Week 1-2)
Phase 2: project.ts, message.ts 拆分 (Week 2-3)
Phase 3: flowchart.ts 拆分 (Week 3-4)
```

#### 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 迁移期间破坏现有功能 | 中 | 高 | 兼容层 + 完整测试覆盖 + 蓝绿部署 |
| 循环依赖引入 | 低 | 高 | 使用 `index.ts` 统一转发，禁止直接跨服务 import |
| 类型不一致 | 中 | 中 | types/ 目录集中管理，统一导出 |
| API 响应时间未达预期 | 低 | 中 | 性能测试验证基准，达标后再合并 |

**残余风险**: 兼容层 `api.ts` 作为技术债遗留，需后续清理计划。

#### 收益/成本比

- **成本**: 3-5 人天 + 测试编写
- **收益**: 可维护性提升、测试覆盖率提升、响应时间降低 20%
- **比值**: ⭐⭐⭐⭐ (高) — 一次投入，长期收益

### 2.2 A-002: 首页五步流程重构

#### 可行性分析

**技术可行性**: ⚠️ **条件可行**（依赖 A-001 完成）

| 评估维度 | 结论 | 依据 |
|----------|------|------|
| 技术复杂度 | 高 | 引入 XState 状态机，流程逻辑重写 |
| 团队能力 | 需确认 | XState 需学习成本，当前无使用经验 |
| 依赖外部 | XState 库 | 需评估对 bundle size 的影响 |
| 用户影响 | 高 | 流程变更影响所有用户，需 A/B 测试 |

**核心前置条件**:
1. ✅ A-001 Phase 1 (auth.ts) 必须完成并验收
2. ⚠️ XState 学习曲线评估（建议 0.5 天 POC）
3. ⚠️ 状态机与现有 React 组件的集成方案需 Architect 确认

**实现路径**（分阶段）:
```
Phase 1 (Week 4-5): 状态机框架搭建，Step 1-2 迁移
Phase 2 (Week 5-6): Step 3-4 新增步骤
Phase 3 (Week 6-7): Step 5-7 及向后兼容
```

#### 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| A-001 延期导致整体延期 | 高 | 高 | 严格控制 A-001 时间盒，每周检查里程碑 |
| XState 学习成本超预期 | 中 | 中 | 提前做 POC，降低不确定性 |
| 用户习惯变更抵触 | 中 | 中 | A/B 测试 + 渐进切换 + 回滚机制 |
| 状态机回退成本极高 | 高 | 中 | 仅在 MVP 验收后合入主分支 |
| 向后兼容实现复杂 | 中 | 中 | 配置化 maxSteps，支持 3/7 步切换 |

**残余风险**: 状态机替换后完全回退成本极高，建议通过 feature flag 控制发布。

#### 收益/成本比

- **成本**: 2-3 人周 + A/B 测试 + 回滚机制开发
- **收益**: 流程可扩展性、用户体验改善（转化率 +15%）、架构现代化
- **比值**: ⭐⭐⭐ (中) — 收益明确但投入大，存在用户适应风险

#### ⚠️ 关键建议

> **暂缓启动条件**: A-001 Phase 1 未验收前，不得开始 A-002 任何开发工作。

### 2.5 P-002: 需求变更追踪系统

#### 可行性分析

**技术可行性**: ⚠️ **有限可行**

| 评估维度 | 结论 | 依据 |
|----------|------|------|
| 技术复杂度 | 中等 | Git 版本控制 + changelog 生成 |
| 团队能力 | 具备 | Git 操作脚本无难度 |
| 需求蔓延风险 | 高 | "变更追踪"范围模糊，容易扩展为完整 Issue 系统 |
| 优先级的合理性 | 低 | 现有 GitHub Issues 已具备基本追踪能力 |

**⚠️ 主要问题**:
1. **范围蔓延**: 需求变更追踪 vs. 完整 Issue 管理系统边界不清
2. **已有替代**: GitHub Issues / Projects 已提供变更追踪能力
3. **ROI 存疑**: 投入产出比低于其他提案

#### 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 需求蔓延为完整系统 | 高 | 高 | 严格限制在 changelog 自动化，不做 Issue 管理 |
| 与现有工具重复建设 | 高 | 中 | 优先评估 GitHub Actions 方案 |
| 开发完成后无人使用 | 高 | 中 | 确认实际使用场景后再开发 |

#### 收益/成本比

- **成本**: 3-4 人天
- **收益**: changelog 自动化，减少人工维护
- **比值**: ⭐⭐ (低) — 建议延期或重新评估范围

#### 决策建议

> ❌ **不建议作为独立项目启动**。建议通过 GitHub Actions + `auto-changelog` 插件实现同等效果（0.5 人天），或合并到 P-001 工具集中。
