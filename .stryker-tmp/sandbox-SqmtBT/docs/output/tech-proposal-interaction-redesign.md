# VibeX 交互重构 - 技术提案与任务拆分

**版本**: v1.0  
**日期**: 2026-03-10  
**状态**: 待评审  
**类型**: 🔴 大型重构

---

## 1. 项目概述

### 1.1 重构目标
将 VibeX 从"黑盒一键生成"改造为"协作式 DDD 建模平台"

### 1.2 核心变化

| 维度 | 现状 | 目标 |
|------|------|------|
| 首页 | Dashboard 项目列表 | 对话入口 + 5步设计流程 |
| 生成方式 | 一键生成 | 分步协作生成 |
| 编辑能力 | 无 | 对话修改 + 代码修改 |
| 导出能力 | 无 | PRD + 静态原型链接 |

### 1.3 影响范围
- 前端: 80% 页面重构
- 后端: 20% API 调整（核心流程不变）
- 数据库: 10% 表结构调整

## 2. 架构调整

### 2.1 页面路由重构

```
现状:
/                    → Dashboard (项目列表)
/confirm             → 需求输入页
/project/:id         → 项目详情（碎片化）

目标:
/                    → 首页（对话入口 + 最近项目）
/design              → 设计流程入口
/design/clarify      → Step 1: 需求澄清
/design/model        → Step 2: 领域模型
/design/flow         → Step 3: 业务流程
/design/ui           → Step 4: UI 结构
/design/prototype    → Step 5: 原型生成
/project/:id         → 项目详情（容器化展示）
```

### 2.2 状态管理重构

```typescript
// 新增全局设计状态
interface DesignState {
  step: 1 | 2 | 3 | 4 | 5;
  
  // Step 1: 需求澄清
  clarification: {
    rawInput: string;
    collected: ClarificationData;
    confirmed: boolean;
  };
  
  // Step 2: 领域模型
  domainModel: {
    entities: Entity[];
    relations: Relation[];
    confirmed: boolean;
  };
  
  // Step 3: 业务流程
  businessFlow: {
    flows: Flow[];
    activeFlowId: string;
    confirmed: boolean;
  };
  
  // Step 4: UI 结构
  uiStructure: {
    pages: Page[];
    activePageId: string;
    confirmed: boolean;
  };
  
  // Step 5: 原型
  prototype: {
    url: string;
    generated: boolean;
  };
}
```

### 2.3 数据流设计

```
用户输入
    ↓
[Zustand Store] designState
    ↓
[API] 提交到后端
    ↓
[D1 Database] 持久化
    ↓
[Cloudflare Workers] 生成原型
    ↓
[Cloudflare Pages] 静态部署
```

## 5. 测试约束

### 5.1 E2E 测试覆盖要求

**每个功能点必须覆盖:**

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| 正向案例 | 正常操作流程 | 输入需求 → 提问 → 回答 → 确认 |
| 反向案例 | 异常/错误处理 | 空输入、网络错误、超时 |
| 边界案例 | 边界条件 | 超长输入、特殊字符、并发操作 |

### 5.2 测试用例模板

```typescript
describe('FEAT-010: 需求输入框', () => {
  // 正向案例
  it('should submit valid input', async () => {
    await page.fill('input', '我想做一个电商小程序');
    await page.click('button[type="submit"]');
    await expect(page.locator('.analyzing')).toBeVisible();
  });
  
  // 反向案例
  it('should show error for empty input', async () => {
    await page.click('button[type="submit"]');
    await expect(page.locator('.error-message')).toContainText('请输入需求');
  });
  
  // 边界案例
  it('should truncate input over 5000 chars', async () => {
    const longText = 'a'.repeat(6000);
    await page.fill('input', longText);
    const value = await page.inputValue('input');
    expect(value.length).toBeLessThanOrEqual(5000);
  });
});
```

### 5.3 验收标准

**"不能只是做了，必须是做到了":**

| 阶段 | 验收标准 |
|------|----------|
| 开发完成 | 功能可用，无阻塞性 Bug |
| 测试通过 | E2E 测试覆盖率 ≥ 80%，正反案例全覆盖 |
| 代码评审 | 无 Critical/High 级别问题 |
| 上线验收 | 生产环境验证通过，用户可用 |

## 6. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 改动范围大，影响现有功能 | 高 | 高 | 分阶段上线，保留旧版入口 |
| AI 生成质量不稳定 | 中 | 高 | 引入评分机制，低分重新生成 |
| 用户不适应新流程 | 中 | 中 | 引导动画、帮助文档 |
| E2E 测试维护成本高 | 低 | 中 | 使用 Page Object 模式 |
