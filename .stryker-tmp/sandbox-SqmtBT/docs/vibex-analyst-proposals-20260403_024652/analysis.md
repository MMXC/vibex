## 提案详情

### A-1: 画布状态引导层 — P1

**分析视角**: 用户体验 + 信息架构

**问题描述**:
当前画布页面缺少 Phase 状态感知：
- 用户进入 editor 后不知道自己在哪个阶段（上下文/流程/组件）
- 三树切换时无视觉锚点
- 新用户首次使用缺乏引导

**竞品参考**:

| 产品 | 方案 | 特点 |
|------|------|------|
| Figma | 顶部 Toolbar 显示当前模式 + 步骤指示器 | 始终可见，渐进式 |
| Notion | 侧边导航高亮当前页 | 上下文清晰 |
| Linear | 左侧状态栏 + 顶部 Breadcrumb | 导航清晰 |

**建议方案**:

方案 A: **Phase 状态栏 + 引导卡片**（推荐）
```
[Phase 指示器]
  ● 上下文 ○ 流程 ○ 组件
  "当前：上下文建模 — 请为系统识别限界上下文"

[引导卡片]（仅首次出现时显示）
  "建议：先定义上下文边界，再展开业务流程"
```

方案 B: **Toast 引导**（次选）
- Phase 切换时显示 Toast 提示

**工作量估算**: 2-3h（前端）

**验收标准**:
- [ ] Phase 指示器在画布顶部始终可见
- [ ] 切换 Phase 时指示器状态同步更新
- [ ] 首次进入画布显示引导卡片（仅一次）
- [ ] 引导卡片可手动关闭且不再出现

### A-2: 设计产物分析指标体系 — P2

**分析视角**: 产品分析 + 数据驱动

**问题描述**:
无任何使用数据，无法回答：
- 用户平均在哪个 Phase 停留最久？
- 哪些 Context/Flow/Component 被使用最多？
- 完成一个项目平均需要多少步？

**建议方案**:

方案 A: **轻量埋点 + 分析仪表盘**（推荐）
```
埋点事件:
  - canvas_phase_entered { phase: 'context' | 'flow' | 'component' }
  - node_created { type, phase }
  - flow_completed { duration_minutes }
  - export_triggered { format }

仪表盘（/analytics）:
  - 项目完成率漏斗
  - 平均节点数/项目
  - Phase 停留时长分布
```

**注意**: 不需要真实用户追踪，仅聚合指标

**工作量估算**: 4-6h（后端 2h + 前端 2-4h）

**验收标准**:
- [ ] `/analytics` 页面可访问
- [ ] 显示最近 7 天项目完成率
- [ ] 显示平均节点数/项目
- [ ] 无个人用户级别追踪（隐私合规）

---

### A-3: 设计交接与导出增强 — P1

**分析视角**: 用户旅程 + 价值实现

**问题描述**:
用户完成 DDD 建模后无明确下一步：
- 导出的 JSON/代码无人接手
- 无法分享给团队成员（只有 owner 可访问）
- 无版本对比能力（无法看到设计演进）

**竞品参考**:

| 产品 | 方案 |
|------|------|
| Figma | 分享链接 + 权限控制 + 版本历史 |
| Miro | 导出 PDF/PNG + 团队协作 |
| Lucidchart | 导出 Visio/PNG + 版本比较 |

**建议方案**:

方案 A: **协作分享 + 版本快照**（推荐）
```
协作:
  - 项目可设置协作者（邮箱邀请）
  - 协作者可评论/编辑
  - 分享链接（只读/可编辑）

版本:
  - 每个设计保存一个快照
  - 快照可命名（"v1.0 初始设计"）
  - 快照之间可比较差异
```

**工作量估算**: 8-12h（后端 4-6h + 前端 4-6h）

**验收标准**:
- [ ] 项目可生成只读分享链接
- [ ] 协作者可通过邮箱接受邀请
- [ ] 设计版本快照支持命名和保存
- [ ] 可查看历史快照内容

---

### A-4: 端内 Feedback 收集机制 — P2

**分析视角**: 用户声音 + 质量闭环

**问题描述**:
用户遇到问题时只能通过外部渠道反馈：
- 问题反馈路径不清晰
- 无问题分类，PM 难以归类
- 无反馈处理状态追踪

**建议方案**:

方案 A: **Floating Feedback 按钮 + 分类表单**（推荐）
```
位置: 画布右下角浮动按钮 [💬 反馈]

表单内容:
  - 问题类型: [Bug / 功能建议 / 体验问题 / 其他]
  - 描述: [必填，200字以内]
  - 当前页面: [自动填充]
  - 截图: [可选上传]
  - 联系方式: [可选]

后端:
  - 存入 feedback 表
  - 自动创建 GitHub Issue（带 label）
  - 通知 PM 频道
```

**工作量估算**: 3-4h

**验收标准**:
- [ ] 反馈按钮在画布页面始终可见
- [ ] 提交后显示确认 Toast
- [ ] PM 频道收到通知
- [ ] 反馈自动带上当前 URL 和 Phase

---

### A-5: 质量仪表盘与趋势追踪 — P2

**分析视角**: 质量工程 + 持续改进

**问题描述**:
每次迭代质量依赖人工验收，无量化指标：
- 无法回答"这次发布比上次稳定吗？"
- E2E 测试通过率无趋势图
- 线上错误率未知

**建议方案**:

方案 A: **CI 质量仪表盘 + 趋势图**
```
指标:
  - E2E 通过率趋势（最近 10 次 CI）
  - 单元测试覆盖率变化
  - TypeScript 错误数趋势
  - Lighthouse 性能分数

集成:
  - GitHub Actions 完成后自动更新
  - 趋势图显示在 /quality 页面
  - 异常时 Slack 报警
```

**工作量估算**: 3-4h（脚本 + 展示页面）

**验收标准**:
- [ ] `/quality` 页面显示 E2E 通过率趋势
- [ ] 显示 TypeScript 错误数趋势
- [ ] 趋势图覆盖最近 10 次构建
- [ ] 通过率 < 90% 时 Slack 报警
