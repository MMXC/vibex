# 需求分析: vibex-api-retry-circuit

**项目**: vibex-api-retry-circuit
**版本**: 1.0
**日期**: 2026-03-15
**分析者**: Analyst Agent

---

## 执行摘要

项目前端使用裸 `fetch()` 调用，无重试和熔断机制。网络抖动直接导致用户请求失败。需求是建立统一 API 客户端，实现指数退避重试和熔断机制，**减少 50% 临时错误影响**。

---

## 1. 业务场景

### 1.1 当前问题

| 问题 | 影响 | 频率 |
|------|------|------|
| 网络抖动导致请求失败 | 用户体验差 | 高 |
| 后端临时不可用 | 整个流程中断 | 中 |
| 请求超时无处理 | 页面卡死 | 中 |
| 无统一错误处理 | 代码重复 | 高 |

### 1.2 目标用户

- 产品用户：减少因网络问题导致的操作失败
- 开发者：统一 API 调用方式，减少重复代码

---

## 2. 技术方案

### 2.1 核心组件

| 组件 | 功能 | 复杂度 |
|------|------|--------|
| ApiClient | 封装 fetch，统一调用接口 | 低 |
| RetryHandler | 指数退避重试 | 中 |
| CircuitBreaker | 熔断器状态机 | 中 |
| TimeoutHandler | 超时控制 | 低 |

### 2.2 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大重试次数 | 3 | 避免 无限重试 |
| 初始重试延迟 | 1000ms | 第一次重试等待 |
| 延迟增长因子 | 2x | 指数退避 |
| 请求超时 | 10s | 单次请求最大时间 |
| 熔断阈值 | 5 次/分钟 | 触发熔断的失败次数 |
| 熔断恢复时间 | 30s | 半开状态等待时间 |

---

## 3. 可行性评估

### 3.1 技术可行性: ✅ 高

- 原生 fetch + AbortSignal.timeout 支持良好
- 熔断器模式成熟，可自实现
- 无需引入额外依赖

### 3.2 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 重试导致延迟增加 | 中 | 限制重试次数，显示加载状态 |
| 熔断误判 | 低 | 配置合理的阈值 |
| 兼容性问题 | 低 | 现代浏览器均支持 |

---

## 4. 验收标准

| 标准 | 测试方法 |
|------|----------|
| 重试 3 次后失败 | mock fetch 抛错，验证调用次数 |
| 指数退避延迟正确 | 记录重试时间戳，验证间隔 |
| 熔断器状态转换正确 | 模拟连续失败，验证 OPEN 状态 |
| 超时触发重试 | mock 超时场景，验证行为 |
