## 一、项目背景与目标

### 1.1 项目目标

基于 VibeX 首页 v4 原型，完成前后端详细设计、接口定义、测试用例设计，输出完整的开发指导文件。

### 1.2 核心价值

- **降低 DDD 门槛**: 将复杂的领域驱动设计流程可视化、智能化
- **即时反馈**: 用户输入需求后实时展示 AI 分析结果
- **快速建项**: 分析结果一键创建为项目

### 1.3 目标用户

| 用户类型 | 特征 | 痛点 |
|----------|------|------|
| 产品经理 | 熟悉业务，不懂技术 | 难以清晰表达系统边界 |
| 开发工程师 | 懂技术，业务理解浅 | 领域模型设计困难 |
| 架构师 | 需要快速验证想法 | 缺乏高效工具 |
| 初创团队 | 资源有限 | 没有专人做设计 |

## 三、技术方案选项

### 3.1 方案A: 渐进增强型（推荐）

**核心理念**: 先实现核心功能，再逐步叠加交互优化。

| 阶段 | 内容 | 工作量 |
|------|------|--------|
| Phase 1 | 布局框架 + 静态UI | 约40% |
| Phase 2 | 状态管理 + 数据绑定 | 约30% |
| Phase 3 | SSE流 + AI展示 | 约20% |
| Phase 4 | 交互增强 + 悬浮模式 | 约10% |

**优点**:
- 每个阶段都可交付可用产品
- 风险分散，单点问题不影响全局
- 便于测试和审查

**缺点**:
- 需要多轮迭代
- 早期版本功能有限

### 3.2 方案B: 完整实现型

**核心理念**: 一次性实现全部32个功能点。

**优点**:
- 用户体验一致
- 减少返工

**缺点**:
- 开发周期长
- 风险集中
- 测试覆盖难度大

**结论**: 推荐方案A，理由：
1. 功能依赖关系清晰，适合分阶段
2. SSE + 状态管理是核心技术风险点，需优先验证
3. Mermaid渲染在历史项目中已多次修复，有经验

## 八、实施建议

1. **Mermaid渲染**: 使用 `mermaid-render-fix-v4` 方案，参考已有的验证测试
2. **Zustand Store**: 拆分 designStore 为多个 slice（ui slice + data slice）
3. **SSE**: 使用 `EventSource` + 自定义重连逻辑，参考 `/api/v1/analyze/stream`
4. **CSS**: 统一使用 tokens.css 变量，禁止硬编码
5. **测试策略**: 优先覆盖 P0 功能点，E2E测试覆盖完整流程
