## 1. Problem Statement

### 1.1 当前痛点

| 痛点 | 影响 | 严重度 |
|------|------|--------|
| 无性能监控 | 线上问题无法及时发现 | 🔴 高 |
| 分支覆盖率仅 35.51% | API 质量风险高 | 🔴 高 |
| 错误无法追踪 | 用户问题难以复现 | 🟡 中 |
| 无性能基线 | 优化无依据 | 🟡 中 |

### 1.2 业务影响

- **用户流失**: 性能问题导致用户流失率增加 15%
- **开发效率低**: 无监控导致问题排查时间长 3x
- **质量风险**: 低覆盖率导致回归问题频发

---

## 2. Proposed Solution

### 2.1 方案概述

**双轨并行**:
1. **APM 监控系统** - 实时性能监控 + 错误追踪
2. **API 覆盖率提升** - 从 35.51% 提升至 70%

### 2.2 技术选型

| 模块 | 选择 | 理由 |
|------|------|------|
| APM 监控 | **Sentry** | 开源、功能完整、社区活跃 |
| 前端错误追踪 | Sentry Browser SDK | 与后端统一 |
| 性能指标 | Web Vitals + Sentry | 行业标准 |
| 覆盖率工具 | Jest + Istanbul | 现有工具链 |

## 4. Core User Stories

### US-1: APM 监控集成

**作为** 开发者
**我想要** 实时监控应用性能和错误
**以便于** 快速发现和定位问题

**验收标准**:
- [ ] Sentry 项目创建完成
- [ ] 前端错误自动上报
- [ ] 性能指标 (Web Vitals) 自动收集
- [ ] 错误聚合和去重
- [ ] Slack 告警集成

**优先级**: P0
**工作量**: 8 小时

---

### US-2: API 模块覆盖率提升

**作为** 开发者
**我想要** API 模块测试覆盖率达到 70%
**以便于** 保证 API 质量和稳定性

**验收标准**:
- [ ] 分支覆盖率 ≥ 70%
- [ ] 所有 API 模块有单元测试
- [ ] 关键路径 100% 覆盖
- [ ] CI 集成覆盖率检查

**优先级**: P0
**工作量**: 14 小时

**作为** 运维人员

## 6. APM 监控设计

### 6.1 监控架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Sentry 监控架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  前端应用                                                    │
│       │                                                      │
│       ▼                                                      │
│  Sentry Browser SDK                                         │
│       │                                                      │
│       ├── 错误捕获 → Error Event                            │
│       ├── 性能监控 → Performance Transaction               │
│       └── 用户行为 → Breadcrumbs                            │
│              │                                               │
│              ▼                                               │
│         Sentry Cloud                                        │
│              │                                               │
│              ├── 事件聚合                                    │
│              ├── 告警规则                                    │
│              └── Slack 通知                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 监控指标

| 类别 | 指标 | 说明 |
|------|------|------|
| **错误监控** | 错误率 | 每小时错误数 |
| | 错误类型 | JS 错误、网络错误、资源错误 |
| | 影响用户数 | 受影响用户百分比 |
| **性能监控** | LCP | 最大内容绘制时间 |
| | FID | 首次输入延迟 |
| | CLS | 累积布局偏移 |
| | TTFB | 首字节时间 |
| **用户体验** | 会话数 | DAU |
| | 页面加载时间 | P50/P95/P99 |

### 6.3 告警规则

| 告警类型 | 条件 | 级别 |
|----------|------|------|
| 错误率突增 | > 基线 50% | Critical |
| 新错误 | 首次出现 | Warning |
| 性能下降 | LCP > 3s | Warning |
| 影响用户 | > 10% 用户 | Critical |

## 10. 附录

### A. Sentry 配置示例

```typescript
// lib/sentry.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.1,
  replaysSessionSampleRate: 0.1,
  replaysOnErrorSampleRate: 1.0,
  integrations: [
    new Sentry.BrowserTracing(),
    new Sentry.Replay(),
  ],
});
```

module.exports = {
