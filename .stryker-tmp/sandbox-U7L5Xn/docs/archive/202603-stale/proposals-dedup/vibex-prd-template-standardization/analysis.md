# 需求分析: PRD 模板标准化

**项目**: vibex-prd-template-standardization
**日期**: 2026-03-17
**分析师**: Analyst Agent

---

## 1. 执行摘要

### 目标

建立标准化的 PRD 模板，包含根因分析章节和验收条件格式，提升需求分析效率，减少开发过程中的需求遗漏。

### 成功指标

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| PRD 返工率 | ~20% | <10% |
| 验收条件明确度 | 无量化 | 100% 可测试 |
| 根因分析覆盖率 | ~30% | 100% (P0/P1) |

### 预期收益

- 需求遗漏减少 50%
- 测试通过率提升 20%
- PM 编写效率提升 30%

---

## 2. 问题定义

### 2.1 核心痛点

| 问题 | 当前状态 | 影响 |
|------|----------|------|
| 根因分析缺失 | 70% PRD 无根因分析章节 | 开发过程中频繁发现遗漏点 |
| 验收条件模糊 | 存在"优化性能"等不可测试条件 | 测试无法编写断言 |
| Epic 粒度不均 | 0.5h~4h 跨度大 | 排期不准确 |
| 版本历史缺失 | 修改无记录 | 需求变更不可追溯 |

### 2.2 业务场景

**场景 1**: PM 编写新功能 PRD
- 当前: 从零开始，结构随意
- 期望: 基于模板快速填充，结构规范

**场景 2**: Reviewer 审查 PRD
- 当前: 无检查清单，审查点随意
- 期望: 有标准化检查清单，逐项核对

**场景 3**: Dev 根据 PRD 开发
- 当前: 验收条件模糊，需反复沟通
- 期望: 每个功能点有明确可测试的验收标准

## 5. 技术方案

### 5.1 模板文件结构

```
/root/.openclaw/vibex/docs/
├── templates/
│   ├── prd-template.md          # PRD 标准模板
│   └── acceptance-criteria-checklist.md  # 验收条件检查清单
├── guides/
│   ├── epic-splitting-guide.md  # Epic 拆分指南
│   └── prd-version-control.md   # 版本控制规范
└── prd/
    └── [现有 PRD 文件]
```

### 5.2 根因分析模板

**四象限格式**:

| 维度 | 内容 |
|------|------|
| **问题描述** | 用户可见的症状 |
| **根因** | 导致问题的根本原因 |
| **影响范围** | 受影响的功能/用户 |
| **解决方案** | 修复方向 |

### 5.3 验收条件格式规范

**格式要求**:
1. 使用 `AC{n}.{m}:` 前缀
2. 必须可写 `expect()` 断言
3. 避免模糊词汇: "优化"、"提升"、"改善"

**示例**:
```markdown
✅ 正确: AC1.1: API 响应时间 < 200ms
❌ 错误: AC1.1: 优化 API 性能

✅ 正确: AC2.1: 点击按钮后显示 Toast 提示 "操作成功"
❌ 错误: AC2.1: 提升用户体验
```

## 6. 验收标准

### 6.1 功能验收

| ID | 验收标准 | 测试方法 |
|----|----------|----------|
| AC1.1 | `docs/templates/prd-template.md` 文件存在 | `test -f` |
| AC1.2 | 模板包含根因分析章节 | `grep "根因分析"` |
| AC1.3 | 模板包含版本历史章节 | `grep "版本历史"` |
| AC2.1 | 验收条件检查清单存在 | `test -f` |
| AC2.2 | 检查清单包含 ≥5 个检查项 | `grep -c "\\[\\]"` |
| AC3.1 | Epic 拆分指南存在 | `test -f` |
| AC3.2 | 指南包含粒度建议(1-2h) | `grep "1-2"` |

### 6.2 质量验收

| 指标 | 目标 | 验证方法 |
|------|------|----------|
| 模板完整性 | 所有章节可填充 | PM 试用反馈 |
| 格式一致性 | Markdown 渲染正常 | 预览检查 |
| 可操作性 | PM 可独立使用 | 新 PRD 编写验证 |
