## 3. 详细实施计划

### E1: Sprint 复盘会议 — 2h

#### Step 1.1: 会议准备（10min）

- [ ] 确认参会人员名单（PM、Dev、Reviewer、Architect 等）
- [ ] 发送日历邀请，确认会议室/视频链接
- [ ] 准备会议议程模板（4 项固定议程）

#### Step 1.2: 主持复盘会议（90min）

**议程安排（固定 4 项）**：

| # | 议程 | 时长 | 主持人 |
|---|------|------|--------|
| 1 | 做得好的实践回顾 | 20min | PM |
| 2 | 需要改进的问题 | 25min | Architect |
| 3 | Sprint 关键指标复盘 | 25min | PM |
| 4 | 下个 Sprint 目标对齐 | 20min | PM |

**会议要求**：
- 至少覆盖上述 4 项议程
- 每人至少提出 1 条做得好的实践
- 每人至少提出 1 条待改进问题
- 会议时长 ≥ 90 分钟

#### Step 1.3: 撰写复盘文档（20min）

- [ ] 创建 `docs/retrospectives/2026-04-01.md`
- [ ] 填充 4 项议程内容
- [ ] 记录做得好的实践（≥ 5 条，GP-1 ~ GP-N 编号）
- [ ] 记录需改进的问题（≥ 3 条，IM-1 ~ IM-N 编号）
- [ ] 会议总结（≥ 500 字）

#### Step 1.4: 文档自检（5min）

- [ ] 验证文件存在
- [ ] 验证好的实践 ≥ 5 条
- [ ] 验证改进项 ≥ 3 条
- [ ] 验证字数 ≥ 500

**产出**: `docs/retrospectives/2026-04-01.md`

### E2: 下个 Sprint 规划 — 3h

#### Step 2.1: 分析现有 Backlog（90min）

- [ ] 收集所有未完成的 Feature/Bug 需求
- [ ] 评估每个需求的工作量和优先级
- [ ] 与 PM 对齐 Epic 划分原则（1 Epic = 独立可交付价值）
- [ ] 确定下个 Sprint 的范围边界

**输入来源**：
- `docs/` 下的未完成需求文档
- Sprint 1 遗留问题
- 复盘会议中提出的改进项（IM-X → 可能转化为 Epic）

#### Step 2.2: 撰写 Sprint PRD（90min）

- [ ] 创建 `docs/sprint-20260402/prd.md`
- [ ] 编写执行摘要（背景、目标、成功指标）
- [ ] 规划 ≥ 3 个 Epic（包含 E1/E2/E3 命名规则）
- [ ] 每个 Epic 填写：
  - 名称和描述
  - 优先级（P0/P1/P2）
  - 工时估算（Xh 格式）
  - 功能点列表（含验收标准）
- [ ] 填写验收标准汇总表（expect() 断言）
- [ ] 编写 DoD（Definition of Done）

**PRD 模板结构**：

```markdown
# PRD: Sprint 2 — [Sprint 名称]

## 执行摘要
## Epic 总览（≥ 3 Epic）
## Epic 1: [名称] — [Xh] — [P0/P1/P2]
  ### 功能点表
## Epic 2: [名称] — [Xh] — [P0/P1/P2]
  ### 功能点表
## Epic 3: [名称] — [Xh] — [P0/P1/P2]
  ### 功能点表
## 验收标准
## DoD
```

**要求**：
- Epic 数量 ≥ 3
- 优先级排序清晰（P0 最先执行）
- 每个 Epic 有工时估算
- 文档字数 ≥ 1000

#### Step 2.3: PRD 自检（5min）

- [ ] 验证文件存在
- [ ] 验证 Epic 数量 ≥ 3
- [ ] 验证 P0/P1/P2 优先级标记存在
- [ ] 验证每个 Epic 有工时估算
- [ ] 验证字数 ≥ 1000

**产出**: `docs/sprint-20260402/prd.md`

### E3: 技术债清理计划 — 1h

#### Step 3.1: 收集技术债清单（30min）

- [ ] 回顾 Sprint 1 中发现但未处理的技术债务
- [ ] 识别三类已知债务：
  - **MSW Mock 问题**：服务worker模拟不稳定
  - **Canvas API 问题**：画布交互逻辑缺陷
  - **Playwright 测试**：E2E 测试覆盖不足
- [ ] 补充其他技术债项

#### Step 3.2: 分配责任人和工时（30min）

- [ ] 为每项债务指定明确责任人（owner）
- [ ] 为每项债务评估工时（Xh 格式）
- [ ] 确定优先级（P0/P1/P2）
- [ ] 创建 `docs/tech-debt/cleanup-plan.md`

**债务清单表模板**：

```markdown
| ID | 标题 | 责任人 | 工时 | 优先级 | 类别 | 影响范围 |
|----|------|--------|------|--------|------|----------|
| TD-1 | [MSW Mock 不稳定] | @owner | 4h | P1 | test | API Mock |
```

**要求**：
- 所有债务必须有 owner（非空）
- 所有债务必须有工时估算
- 优先级排序（P0 → P1 → P2）
