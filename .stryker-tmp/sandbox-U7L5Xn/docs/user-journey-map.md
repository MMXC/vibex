# 用户旅程图

## 目标用户

| 用户画像 | 特征 | 痛点 |
|---------|------|------|
| 前端开发者 | 有代码能力，追求效率 | 手写重复组件耗时，希望快速拿到可运行代码 |
| 产品经理 | 懂需求表达，代码能力弱 | PRD 到原型转换成本高，沟通成本大 |
| 创业团队 | 资源有限，速度优先 | 需要快速验证想法，没有专职设计师 |
| 全栈工程师 | 需要前后端协同 | 前后端接口定义与组件开发脱节 |

## 用户旅程

### 主流程

```
用户输入需求 ──→ AI 澄清问题 ──→ 生成领域模型
      │                              │
      │                              ▼
      │                       生成业务流程
      │                              │
      ▼                              ▼
预览 + 调整 ◄─── 生成组件代码 ◄──────┘
      │
      ▼
导出代码
```

### 详细阶段

| 阶段 | 用户行为 | VibeX 响应 | 用户情绪 |
|------|---------|-----------|---------|
| 1. 需求输入 | 粘贴/输入 PRD 或描述需求 | 解析需求文本，识别关键实体 | 中性 → 期待 |
| 2. AI 澄清 | 回答追问 | 生成 3-5 个澄清问题 | 专注 → 信任 |
| 3. 领域建模 | 审阅/确认领域模型 | 生成 ER 图和领域类 | 好奇 → 惊喜 |
| 4. 流程生成 | 审阅业务流程 | 生成流程图和状态机 | 认可 → 满意 |
| 5. 组件生成 | 预览生成的组件 | 输出可运行代码 | 兴奋 → 期待 |
| 6. 调整优化 | 提出修改意见 | 迭代更新代码 | 参与 → 满足 |
| 7. 导出使用 | 复制/下载代码 | 导出完整项目代码 | 成就 → 推荐 |

## 关键场景

### 场景 1: 快速 MVP 构建

**用户**: 创业者张明，需要验证社区产品想法

**旅程**:
1. 输入: 「我想做一个问答社区，用户可以提问、回答、点赞」
2. VibeX 追问: 「社区是否需要版块分类？是否需要积分系统？」
3. 生成: 用户/问题/回答/点赞 领域模型
4. 生成: 提问流程、回答流程、积分流程
5. 生成: 问题列表组件、回答表单组件、点赞按钮组件
6. 预览: 实时查看 UI 效果
7. 导出: 完整 React 项目代码

**关键触点**: AI 追问环节 → 减少后续返工

**用户评价**: 「30 分钟拿到可运行的社区原型」

### 场景 2: DDD 领域建模

**用户**: 架构师李娜，需要为微服务拆分提供领域边界参考

**旅程**:
1. 输入: 完整的业务需求文档
2. VibeX 解析: 识别聚合根、实体、值对象
3. 生成: UML 类图和限界上下文
4. 生成: 领域事件列表
5. 导出: PlantUML 或 Mermaid 格式

**关键触点**: 领域模型可视化 → 团队对齐

**用户评价**: 「DDD 建模时间从 3 天缩短到 3 小时」

### 场景 3: PRD 可视化

**用户**: 产品经理王芳，向开发团队传达需求

**旅程**:
1. 输入: 原始 PRD 文档
2. VibeX 理解: 提取关键流程和页面
3. 生成: 交互式原型（可点击跳转）
4. 生成: 页面流程图
5. 分享: 生成可分享链接

**关键触点**: 原型预览 → 开发团队理解一致

**用户评价**: 「不用反复开会解释需求了」

## 优化建议

| 阶段 | 优化方向 | 预期收益 |
|------|---------|---------|
| 需求输入 | 支持 Markdown/Word/截图多格式 | 降低输入门槛 |
| AI 澄清 | 提供澄清模板，用户可选择 | 加速沟通效率 |
| 领域建模 | 支持手动调整模型关系 | 提高准确性 |
| 流程生成 | 支持 BPMN 标准格式导出 | 兼容企业流程 |
| 组件生成 | 支持 Vue/React/Svelte 多框架 | 扩大用户覆盖 |
| 导出使用 | 支持一键部署到 Vercel/Netlify | 消除最后一步 |
