## 1. 业务场景分析

### 1.1 VibeX 目标用户

| 用户群体 | 画像描述 | 核心诉求 |
|---------|---------|---------|
| **产品设计师** | 需要快速将需求转化为可交互原型 | 高效、所见即所得 |
| **全栈工程师** | 使用 AI 辅助开发原型设计工具 | 可维护、可扩展 |
| **AI 开发者** | 关注 prompt 工程和 AI 生成质量 | 稳定的 AI 输出和错误处理 |
| **团队协作** | 设计-开发协作场景 | 可共享、可版本控制 |

### 1.2 核心价值

VibeX 是 AI 驱动的原型设计工具，核心价值链路：
```
需求输入 → 限界上下文 → 领域模型 → 业务流程 → 组件树
```

技术债务风险集中在**组件树层**（三个树各自维护状态）和**渲染层**（2175行 renderer.ts），直接影响 AI 生成结果的稳定性和可维护性。

### 1.3 当前技术债务现状

| 风险维度 | 严重程度 | 影响范围 |
|---------|---------|---------|
| 超长文件（7个 >800行） | 🔴 高 | 可维护性、测试覆盖率 |
| 类型安全问题（8个文件 `as any`） | 🟡 中 | 运行时错误风险 |
| 组件目录混乱（70个目录） | 🟡 中 | 新人上手、代码复用 |
| 三树状态碎片化 | 🔴 高 | Canvas 性能、状态一致性 |
| 错误边界缺失 | 🟡 中 | 用户体验、错误定位 |

## 2. 核心 JTBD（Jobs To Be Done）

### JTBD 1: 降低认知负荷，实现可靠代码维护
> **用户**: 全栈工程师 / AI 开发者
>
> "作为代码维护者，我希望每个文件的职责单一、行数可控（<300行），这样我能快速定位问题、理解逻辑，而不会被 2000 行的文件压垮认知资源。"
>
> **触发信号**: 当新增功能需要修改 `renderer.ts` 或 `canvasStore.ts` 时，工程师需要花 30 分钟以上理解上下文才能安全修改。

### JTBD 2: 保证 Canvas 操作流畅性
> **用户**: 产品设计师
>
> "作为设计师，我在拖拽画布节点时希望操作流畅无卡顿（60fps），而不是每次拖动都要等待 200-500ms 的渲染延迟。"
>
> **触发信号**: 当前三树各自维护状态 + 无虚拟化，长列表场景（50+ 节点）下出现明显卡顿。

### JTBD 3: 实现优雅的错误处理和降级
> **用户**: 所有用户
>
> "当某个组件出错时，我只希望出错的部分展示错误信息，而不是整个页面崩溃白屏。"
>
> **触发信号**: AI 生成组件失败时，当前会导致整个 Canvas 页面崩溃，用户体验极差。

### JTBD 4: 建立可扩展的组件架构
> **用户**: 团队 / AI 开发者
>
> "作为团队成员，我希望组件目录结构清晰、命名规范，这样我能快速找到需要的组件，而不是在 70 个目录中迷失。"
>
> **触发信号**: 新人入职需要 1 周以上才能熟悉组件结构，AI 生成代码时频繁出现重复组件。

### JTBD 5: 确保类型安全，防止运行时错误
> **用户**: 全栈工程师
>
> "作为开发者，我希望 TypeScript 能捕获尽可能多的类型错误，这样我可以减少运行时调试时间，专注于业务逻辑。"
>
> **触发信号**: 8 个文件使用 `as any`，AI 生成代码时类型推断失效。

## 3. 技术方案选项

### 方案 A: 激进重构优先（P0 驱动）

**核心理念**: 先解决最严重的架构问题（P0），其他提案顺延。

| 阶段 | 提案 | 工时 | 说明 |
|------|------|------|------|
| Phase 1 | renderer.ts 拆分 | 2天 | 入口+工厂 → 按端拆分 → 按元素类型拆分 |
| Phase 2 | canvasStore.ts 拆分 | 1.5天 | 显式 slice 分离 + middleware 独立 |
| Phase 3 | CanvasPage.tsx 组件拆分 | 1.5天 | 复合组件模式，CanvasToolbar/Viewport 独立 |
| Phase 4 | 三树状态统一 | 2天 | Zustand 单一 store，React.memo 优化 |
| Phase 5 | 虚拟化列表 | 1天 | react-virtual 集成 |
| Phase 6 | 错误边界增强 | 1天 | 按路由添加 error.tsx |
| Phase 7 | 类型安全清理 | 1天 | 渐进式清理 8 个文件 |
| Phase 8 | 组件目录治理 | 3天 | 70 → 5 大类重组 |
| Phase 9 | 依赖安全审计 | 0.5天 | npm audit + Dependabot |

**总工时**: 约 14.5 天（按 1 人）

**优点**:
- 一次性解决核心架构问题，后续迭代轻量
- 代码审查效率显著提升

**缺点**:
- 周期长（约 3 周），期间可能有其他紧急需求打断
- 重构风险高，需要充分测试覆盖
