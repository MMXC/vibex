## 提案 1: OpenViking API Gateway 重构

**提案方**: Architect | **优先级**: P1 | **工时**: 4h (Phase 3 可选)

### 🎯 战略价值评估 (CEO 视角)

| 维度 | 评分 | 说明 |
|------|------|------|
| 问题真实性 | ⭐⭐⭐⭐⭐ | JSON 并发损坏是真实痛点，已多次出现 |
| 根因 vs 症状 | ⭐⭐⭐⭐ | 解决根本架构问题，不只是打补丁 |
| ROI | ⭐⭐⭐ | 4h 工时，换来长期稳定性，值得 |
| 可逆性 | ⭐⭐⭐⭐ | Phase 1 只加 API，不改调用方式 |
| 创新性 | ⭐⭐ | 标准的 API 封装模式，无新奇之处 |

**判断**: ✅ 值得做

### 🔧 工程可行性评估 (Eng 视角)

| 维度 | 评分 | 说明 |
|------|------|------|
| 复杂度 | 🟡 中等 | 1 个新文件 + 多处调用改造 |
| 依赖 | 🟢 低 | 纯 Python，无外部依赖 |
| 测试覆盖 | 🟡 需补充 | 提案本身缺少测试计划 |
| 风险 | 🟡 中 | Phase 1 破坏现有 agent 的风险需监控 |
| 并发安全 | 🟡 待验证 | JSON 层仍有并发问题，Phase 3 才解决 SQLite |

**工程问题**:
1. Phase 1 用 JSON 作为存储层，并发覆盖问题没有彻底解决，只是加了 API 封装
2. 缺少对现有 12+ agent 调用方式的清点，改造范围不明确
3. 测试计划缺失，只写了验收标准，没写具体测试用例

### 📊 综合评分

| 维度 | 得分 |
|------|------|
| 战略价值 | 4/5 |
| 工程可行性 | 3/5 |
| **综合** | **7/10** |

**建议**: P1，做。但 Phase 1 改为**直接上 SQLite**，不要分三阶段。JSON API 封装是假解法，Phase 3 才能真正解决并发问题。4h 内完全可以直接用 SQLite + 简单的 CRUD API。
