# Architecture Design: vibex-frontend-build-fail

**项目**: vibex-frontend-build-fail  
**任务**: design-architecture  
**架构师**: Architect Agent  
**日期**: 2026-03-20  
**根因**: `zustand` 未在 `package.json` 中声明，依赖 reactflow 传递依赖  

---

## 1. 问题分析

### 1.1 根因

```
package.json (无 zustand!)
    └── reactflow@11.11.4 (依赖 zustand@4.5.7)
            └── zustand@4.5.7 (传递依赖，版本由 reactflow 控制)
                    └── 20 个 store 文件 (直接 import)
```

- 20 个 `.ts/.tsx` 文件直接 `import { create } from 'zustand'`
- `zustand` 未写入 `package.json` 的 `dependencies`
- pnpm 提升传递依赖到顶层 `node_modules` → 本地构建成功
- npm `npm ls` 报告 `extraneous` 警告
- CI/CD 使用 `npm ci` 时可能失败（严格依赖安装）

### 1.2 风险等级

| 风险 | 等级 | 说明 |
|------|------|------|
| 本地开发构建失败 | 🟢 低 | pnpm 传递依赖提升机制保护 |
| CI/CD 构建失败 | 🟡 中 | `npm ci` 严格安装，不含 zustand |
| 版本被 reactflow 绑架 | 🟡 中 | reactflow 未来可能变更 zustand 版本 |

## 2. 技术方案

### 2.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 推荐 |
|------|------|------|------|------|
| **方案 A** | 添加到 `dependencies` | 明确声明，版本可控，CI 兼容 | 多一个依赖声明 | ✅ |
| 方案 B | 添加到 `devDependencies` | 快速 | 语义错误，运行时依赖不应放 dev | ❌ |
| 方案 C | 不修复 | 无 | 隐患持续，CI/CD 风险 | ❌ |

### 2.2 最终决策

**方案 A**：在 `package.json` 的 `dependencies` 中明确声明 `zustand@^4.5.7`

**理由**：
1. `zustand` 是运行时依赖，正确放入 `dependencies`
2. 版本 `^4.5.7` 与 reactflow 当前版本兼容
3. 消除 extraneous 警告，CI/CD 友好
4. 版本控制权从 reactflow 转交给项目维护者

// tests/zustand-dependency.test.ts

describe('zustand dependency', () => {
  it('zustand should be declared in package.json dependencies', () => {
