# 分析报告：Vibex 流程简化（5步→3步）

> 基于 analyst 提案整理 | 日期：2026-03-23

## 📋 需求背景

当前 VibeX 使用 5 步 DDD 建模流程，但存在以下问题：
- 领域模型步骤对用户过于技术化（暴露 DDD 专业术语）
- 流程冗长，用户需要 5 步才能完成一次建模
- 限界上下文和业务流程分离，用户体验割裂

## 🔍 问题分析

### 用户体验问题
1. **术语门槛高**：限界上下文、聚合根等概念对非 DDD 专家的用户不友好
2. **步骤冗余**：5 步流程中，领域模型和限界上下文有大量重叠信息
3. **缺少交互**：用户只能看不能改，无法真正参与建模

### 技术问题
1. 现有数据结构未区分「核心域/支撑域/通用域」
2. 状态管理未覆盖「用户勾选/修改」状态
3. 无 PageComponent 数据结构

## 🎯 解决方案

### 核心思路
- 删除领域模型步骤（改为 AI 生成限界上下文时内嵌）
- 将限界上下文和业务流程合并为 Step1
- 用业务语言替代 DDD 术语

### 方案优点
- 用户体验：从 5 步压缩到 3 步
- 学习成本：无需理解 DDD 专业术语
- 参与感：用户可修改、勾选、添加节点

## ⚠️ 技术风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| AI 生成准确性 | 高 | 建立 Prompt 模板，限制生成范围 |
| 性能问题（大图） | 中 | 虚拟化渲染，分页加载 |
| 向后兼容 | 中 | 保留现有 API，新旧流程共存 |

## ✅ 验收标准
- Step1 能并行生成限界上下文 + 业务流程图
- 用户能修改节点名称、勾选功能点
- Step3 能生成带组件勾选框的页面卡片
- 术语正确转换（限界上下文→业务领域，核心域→核心业务）
