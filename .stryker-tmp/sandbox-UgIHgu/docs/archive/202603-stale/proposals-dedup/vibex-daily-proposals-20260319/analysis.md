## 3. 需求分析

### 3.1 API 服务层拆分 (A-001)

**问题根源**:
- 单文件 api.ts 耦合度过高
- 无法按需加载和独立测试
- 修改影响范围不可控

**技术方案**:
```
src/services/
├── api.ts      # 兼容层
├── auth.ts     # 认证服务
├── project.ts  # 项目管理
├── message.ts  # 消息服务
├── flowchart.ts # 流程图服务
└── index.ts    # 统一导出
```

**验收标准**:
- [ ] 各服务独立运行测试通过率 100%
- [ ] API 响应时间降低 20%
- [ ] 兼容层支持现有调用方式

**风险**: 迁移期间可能破坏现有功能  
**缓解**: 完整测试覆盖 + 兼容层

---

### 3.2 首页五步流程重构 (A-002)

**问题根源**:
- 3步流程无法有效引导复杂需求
- 领域边界未提前定义
- 用户对中间结果缺乏可见性

**新增步骤**:
- Step 3: 需求澄清 (Requirement Clarification)
- Step 4: 限界上下文定义 (Bounded Context Definition)

**依赖关系**:
```
A-001 (API拆分)
    ↓
A-002 (五步流程) 依赖 Step 数据结构
```

**验收标准**:
- [ ] 新流程支持最多 7 步可配置
- [ ] 用户输入转化率提升 15%
- [ ] 向后兼容现有 3 步流程

**风险**: 用户习惯变更  
**缓解**: A/B 测试 + 渐进切换

---

### 3.3 RCA CLI 工具 (A-003)

**问题根源**:
- 问题排查依赖人工日志分析
- 缺乏系统性工具支持
- RCA 流程不标准

**技术方案**:
```bash
rca --log-dir ./logs --date 2026-03-19 --severity error
```

**核心功能**:
| 功能 | 描述 |
|------|------|
| 日志聚合 | 按时间/严重度/模块聚合 |
| 异常模式识别 | 检测重复异常模式 |
| 关联分析 | 跨服务 trace_id 关联 |
| 报告生成 | Markdown 格式分析报告 |

**验收标准**:
- [ ] 支持主流日志格式解析
- [ ] 单次分析时间 < 30秒
- [ ] 报告包含完整 trace 链路

**优势**: 独立 CLI，无主系统依赖，可快速迭代
