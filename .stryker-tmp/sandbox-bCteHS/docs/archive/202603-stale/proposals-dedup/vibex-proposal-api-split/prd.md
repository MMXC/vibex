# PRD: API服务拆分

## 执行摘要

| 属性 | 值 |
|------|-----|
| 项目名 | API服务拆分 |
| 优先级 | P0 |
| 预估工期 | 5-7 天 |
| 负责人 | PM |

### 背景

当前VibeX使用单体API架构，AI生成和文件处理服务可能导致性能瓶颈。API服务拆分可提升系统可扩展性和性能。

---

## Success Metrics

| 指标 | 目标 |
|------|------|
| API架构 | 单体 → 微服务化 |
| 扩展性 | 低 → 高 |
| 响应时间 | 降低 |

---

## 功能需求

### F1: API Gateway 部署

| ID | 功能描述 | 验收标准 |
|----|----------|----------|
| F1.1 | 统一入口配置 | Nginx/Cloudflare 网关正常响应 |
| F1.2 | 路由分发 | 请求按路径分发到对应服务 |
| F1.3 | 健康检查 | Gateway 健康检查接口正常 |

### F2: AI服务拆分

| ID | 功能描述 | 验收标准 |
|----|----------|----------|
| F2.1 | 独立AI服务部署 | Cloudflare Workers 运行 |
| F2.2 | 异步处理 | AI请求异步处理，不阻塞主流程 |
| F2.3 | 结果回调 | 完成后回调通知 |

### F3: 文件服务拆分

| ID | 功能描述 | 验收标准 |
|----|----------|----------|
| F3.1 | 文件上传独立服务 | 文件走独立服务处理 |
| F3.2 | 大文件分片 | 支持分片上传 |
| F3.3 | 进度追踪 | 上传进度实时显示 |

### F4: 知识库服务

| ID | 功能描述 | 验收标准 |
|----|----------|----------|
| F4.1 | 独立查询接口 | 知识库查询走独立服务 |
| F4.2 | 缓存优化 | Redis 缓存加速查询 |

---

## Epic 拆分

| Epic | 描述 | Story |
|------|------|-------|
| Epic 1 | API Gateway | S1.1: 网关部署 / S1.2: 路由配置 / S1.3: 健康检查 |
| Epic 2 | AI服务 | S2.1: 服务部署 / S2.2: 异步处理 / S2.3: 回调机制 |
| Epic 3 | 文件服务 | S3.1: 服务部署 / S3.2: 分片上传 / S3.3: 进度显示 |
| Epic 4 | 知识库服务 | S4.1: 查询接口 / S4.2: 缓存优化 |

---

## 非功能需求

- **性能**: 响应时间降低 30%
- **可用性**: 各服务独立运行，单服务故障不影响整体
- **扩展性**: 支持按需扩展各服务
