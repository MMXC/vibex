# 领域模型全流程检查修复v2需求分析文档

## 1. 业务场景分析
### 1.1 背景
- 2026-03-16已完成领域模型渲染基础修复（commit 005279b），解决了限界上下文生成后点击「生成领域模型」图表不渲染问题
- 修复后存在遗漏问题：缺少空值保护措施，全流程切换一致性未验证
- 目标用户：所有使用vibex DDD设计功能的产品经理、架构师、开发人员
- 核心价值：确保DDD全流程（限界上下文→领域模型→业务流程）稳定可用，无崩溃、无状态不一致问题

### 1.2 核心问题清单
1. **遗漏空值保护**：领域模型渲染逻辑中未对mermaidCode、流数据、组件props等进行空值校验，极端场景下可能导致页面崩溃
2. **组件切换一致性问题**：
   - 限界上下文页面切换到领域模型页面时，状态未正确同步
   - 领域模型页面切换到业务流程页面时，状态传递可能丢失
   - 全局状态管理中context/model/flow三个模块状态不一致

### 1.3 Jobs-To-Be-Done (JTBD)
1. 用户在限界上下文页面生成上下文后，点击切换到领域模型页面，能够看到正确的领域模型图表，无空白或崩溃
2. 用户在领域模型页面生成模型后，点击切换到业务流程页面，能够看到基于领域模型生成的业务流程，数据完整
3. 用户在三个页面之间来回切换时，已生成的内容不会丢失，状态保持一致
4. 当后端返回数据为空、网络错误、生成失败时，页面显示友好的错误提示，不会崩溃

## 2. 技术方案选项
### 方案A：分层空值保护 + 全局状态同步
**实现思路**：
1. **空值保护层**：
   - 组件层：所有接收mermaidCode、模型数据的组件添加`?.`可选链、默认值、fallback UI
   - Hook层：useDDDStream等自定义Hook添加空值校验，异常时返回默认空结构
   - 状态层：Redux slice的reducer中添加输入校验，避免非法数据写入store
2. **状态同步层**：
   - 统一context/model/flow三个slice的状态更新逻辑，添加跨slice状态同步中间件
   - 页面切换时触发状态校验和同步钩子
   - 路由切换时添加状态持久化和恢复逻辑

**优点**：
- 防护全面，从底层到上层都有保护
- 状态同步逻辑集中，易于维护
- 对现有代码侵入性适中

**缺点**：
- 需要编写中间件，开发量中等
- 状态同步逻辑需要考虑并发场景

### 方案B：组件级防护 + 路由切换校验
**实现思路**：
1. **空值保护**：仅在页面组件和渲染组件层添加空值校验和fallback
2. **状态同步**：在路由切换时（限界上下文→领域模型→业务流程）添加状态校验逻辑，缺失则提示用户重新生成
3. 不修改底层状态管理逻辑

**优点**：
- 开发量小，实现快速
- 风险低，不影响现有状态逻辑

**缺点**：
- 防护不全面，底层状态异常时仍可能出现问题
- 用户体验稍差，可能需要重新生成内容

## 3. 可行性评估
### 方案选择：方案A（推荐）
- 优先级：修复空值保护为P0，状态同步为P1
- 预计工时：空值保护（4h） + 状态同步（8h） = 总工时12h
- 技术可行性：现有Redux架构支持中间件扩展，代码结构清晰，实现难度低
- 业务可行性：不影响现有功能使用，修复后用户体验提升明显

## 4. 初步风险识别
| 风险项 | 概率 | 影响 | 应对措施 |
|--------|------|------|----------|
| 状态同步中间件导致现有状态逻辑异常 | 中 | 高 | 编写单元测试覆盖所有状态更新场景，灰度验证 |
| 空值保护过度导致正常数据渲染失败 | 低 | 中 | 保留原有逻辑，仅添加额外校验，不修改正常流程 |
| 路由切换时状态丢失 | 中 | 中 | 添加状态本地缓存（sessionStorage）作为兜底 |
