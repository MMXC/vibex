### 产品体验缺口（PM 视角）

从产品经理角度，Sprint 3 执行期间识别出以下尚未被覆盖的体验缺口：

1. **新用户首次使用门槛高** — 无引导流程，用户不知道从哪里开始
2. **模板能力缺失** — 无预设模板，每次从空白画布开始，效率低
3. **导出体验割裂** — 各 Phase 导出入口分散，无统一交付中心
4. **项目首页无浏览入口** — 无法浏览历史项目，快速开始新项目路径不清晰
5. **快捷键无个性化** — Power User 无法自定义快捷键，使用效率受限

## 技术方案选项

### P-001: 新手引导流程

#### 方案 A: 渐进式引导卡片 + 里程碑激励（推荐）

```
首次访问流程:
  Step 1: 引导卡片 "欢迎使用 VibeX！" [开始引导] [跳过]
  Step 2: 画布提示 "添加你的第一个限界上下文" → 高亮添加按钮
  Step 3: 输入上下文名称 → 完成第一个节点
  Step 4: 里程碑徽章 "✓ 完成第一步" → 继续引导
  Step 5: 流程树引导 "现在添加业务流程"
  Step 6: 完成引导 → 显示 "🎉 你已完成基础建模！"

里程碑徽章:
  - "上下文探索者" — 添加第一个上下文
  - "流程设计师" — 创建第一个流程
  - "组件架构师" — 完成组件树
  - "DDD 入门" — 完成全部三个 Phase
```

**前端组件**:
- `NewUserGuide.tsx` — 引导流程控制器
- `GuideTooltip.tsx` — 单步提示气泡
- `MilestoneBadge.tsx` — 成就徽章

**技术实现**:
- 引导状态存储在 `localStorage`，首次完成后记录
- 引导步骤通过 Zustand store 管理 (`useGuideStore`)
- 高亮遮罩使用 `pointer-events: none` + 绝对定位裁剪

**工时估算**: 5-7h（前端 4h + 测试 1h + 验收 2h）

#### 方案 B: 分步骤视频 + 静态引导页

**优点**: 内容丰富，可图文并茂
**缺点**: 开发成本高，维护成本高，用户无法互动
**工时估算**: 8-10h（需要制作视频/截图内容）

### P-002: 项目模板库

#### 方案 A: JSON 模板 + 模板选择器（推荐）

```
模板数据结构 (templates/*.json):
  {
    "id": "ecommerce",
    "name": "电商系统",
    "description": "适合电商平台领域建模",
    "thumbnail": "/thumbnails/ecommerce.png",
    "category": "业务系统",
    "contexts": [
      { "name": "商品域", "description": "商品目录和库存管理" },
      { "name": "订单域", "description": "订单处理和履约" },
      { "name": "用户域", "description": "用户账户和会员" }
    ],
    "flows": [
      { "name": "下单流程", "context": "订单域" }
    ],
    "tags": ["电商", "标准", "入门"]
  }

用户流程:
  新建项目 → 模板选择器（网格视图）→ 选择模板 → 确认 → 自动创建项目结构
```

**UI 组件**:
- `TemplateSelector.tsx` — 模板选择器（已有，可扩展）
- `TemplateCard.tsx` — 模板卡片（缩略图 + 名称 + 描述）
- `TemplatePreview.tsx` — 模板预览弹窗

**技术实现**:
- 模板存储在 `public/templates/` JSON 文件
- 模板预览使用 `iframe` 沙箱隔离
- 项目创建时 deep clone 模板 JSON 到新项目

**工时估算**: 6-8h（模板制作 2h + 选择器扩展 2h + 创建逻辑 2h）

#### 方案 B: AI 生成模板（远期）

**优点**: 模板数量无上限，个性化
**缺点**: 依赖 AI API，质量不稳定，当前阶段不适合
**工时估算**: 15h+

### P-003: 统一交付中心

#### 方案 A: Tab 聚合交付面板（推荐）

```
页面结构 (/canvas/delivery):
  [Tab: 限界上下文] [Tab: 流程文档] [Tab: 组件清单] [Tab: PRD]

限界上下文 Tab:
  - 上下文列表（名称、描述、节点数）
  - 导出按钮: [JSON] [Markdown] [PlantUML]
  - 批量导出: [导出全部]

流程文档 Tab:
  - 流程列表（名称、步骤数）
  - 导出按钮: [BPMN JSON] [Markdown 步骤说明]
  - 预览面板

组件清单 Tab:
  - 组件树列表（名称、类型、引用数）
  - 导出: [TypeScript 接口] [JSON Schema]

PRD Tab:
  - 自动生成 PRD 大纲
  - 导出: [Markdown] [飞书文档格式] [Word]
```

**技术实现**:
- 新建 `/canvas/delivery` 路由（独立页面）
- 各 Tab 内容懒加载，减少首屏负担
- 导出功能复用现有 `ExportMenu` 组件

**工时估算**: 8-10h（新建页面 3h + 各 Tab 实现 5h + 导出适配 2h）

#### 方案 B: 分散式导出入口

**优点**: 改动小
**缺点**: 用户不知道去哪里找导出入口，导出体验不连贯
**工时估算**: 2h（但用户体验差）

### P-004: 项目浏览体验优化

#### 方案 A: 增强型项目首页（推荐）

```
首页布局优化:
  [Hero 区域]
    "快速开始" — 大型按钮
    "创建新项目" / "从模板创建"

  [最近项目 — 横向滚动卡片]
    项目卡片: 缩略图 + 名称 + 最后修改时间 + Phase 进度
    悬停显示: 快速操作 [打开] [复制] [删除]

  [项目分类 — 标签过滤]
    全部 / 进行中 / 已完成 / 已归档

  [空状态]
    "还没有项目" + 快速创建引导

列表视图切换:
  网格视图（默认）/ 列表视图（紧凑）
  排序: 最近修改 / 名称 / 创建时间
```

**技术实现**:
- 复用现有 `ProjectBar` 组件扩展
- 列表使用 `IntersectionObserver` 懒加载
- 项目缩略图使用 Canvas 截图（`html2canvas`）

**工时估算**: 5-6h（UI 增强 3h + 列表逻辑 2h）

#### 方案 B: 独立项目管理工作台

**优点**: 功能更完整，可扩展性强
**缺点**: 改动范围大，需要重新设计
**工时估算**: 12h+

### P-005: 快捷键个性化配置

#### 方案 A: 本地快捷键配置（推荐）

```
设置页面 (/settings/shortcuts):
  - 快捷键分类: [导航] [编辑] [视图] [Phase 切换]
  - 每行: [操作名称] [当前快捷键] [点击编辑] [重置默认]

  编辑模式:
    - 点击"编辑" → 输入框等待按键
    - 冲突检测: 已占用的快捷键显示警告
    - 保存到 localStorage

  快捷键使用统计:
    - 显示每个快捷键的使用频率
    - 高频快捷键置顶显示
    - 低频快捷键可考虑禁用
```

**技术实现**:
- 新建 `useKeyboardShortcuts` hook（可配置版）
- 配置存储在 `localStorage['vibex-shortcuts']`
- 冲突检测在 `keydown` 监听器中前置检查

**工时估算**: 4-5h（配置 UI 2h + 可配置 hook 2h）

#### 方案 B: 云端快捷键配置（远期）

**优点**: 多设备同步
**缺点**: 需要用户体系，当前阶段优先级低
**工时估算**: 8h+

## 初步风险识别

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| P-001 引导覆盖其他 UI 元素 | 中 | 阻断用户操作 | 引导层 `z-index` 最高，点击跳过立即退出 |

| P-003 交付中心导出格式不统一 | 中 | 导出内容缺失 | 统一导出数据层，各 Tab 调用同一接口 |
| P-004 首页改版影响现有用户习惯 | 低 | 用户需要重新适应 | 提供视图切换，保持默认行为不变 |
| P-005 快捷键冲突检测复杂 | 中 | 冲突提示不准确 | 按键监听器中统一检测，不依赖状态机 |
