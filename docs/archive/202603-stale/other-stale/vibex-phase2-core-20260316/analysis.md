## 二、需求分析

### 2.1 数据库集成 (Supabase)

#### 2.1.1 业务场景

**场景 1: 数据迁移**
> 现有 D1 数据库存储了用户、项目、消息等核心数据。迁移到 Supabase 后，需要确保数据完整性和应用可用性。

**场景 2: 实时数据同步**
> Supabase 提供实时订阅能力，用户可以看到其他协作者的实时变更，无需手动刷新。

**场景 3: 认证集成**
> Supabase 提供内置认证服务，可以简化当前 JWT + KV 的认证方案。

#### 2.1.2 技术方案

```
┌─────────────────────────────────────────────────────────────┐
│                    Architecture Overview                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐    ┌──────────────────┐    ┌────────────┐ │
│  │   Frontend  │───▶│  Cloudflare      │───▶│  Supabase  │ │
│  │  (Next.js)  │    │  Workers (API)   │    │ PostgreSQL │ │
│  └─────────────┘    └──────────────────┘    └────────────┘ │
│        │                    │                      │        │
│        │                    │                      │        │
│        └────────────────────┼──────────────────────┘        │
│                             │                               │
│                    ┌────────▼────────┐                      │
│                    │ Supabase Realtime│                     │
│                    │  (WebSocket)     │                     │
│                    └─────────────────┘                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**迁移策略**：
1. **双写阶段**：新数据同时写入 D1 和 Supabase
2. **回填阶段**：历史数据迁移到 Supabase
3. **切换阶段**：读写全部切换到 Supabase
4. **下线阶段**：停止 D1 写入，保留只读备份

**数据模型映射**：

| D1 表 | Supabase 表 | 迁移复杂度 |
|-------|-------------|-----------|
| User | auth.users + public.profiles | 中（需拆分） |
| Project | public.projects | 低 |
| Message | public.messages | 低 |
| FlowData | public.flows | 低 |
| Agent | public.agents | 低 |
| Page | public.pages | 低 |
| PrototypeCollaboration | public.collaborations | 低 |

#### 2.1.3 技术风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 数据迁移丢失 | 低 | 高 | 双写验证、校验和比对 |
| 性能下降 | 中 | 中 | 连接池优化、索引优化 |
| 认证兼容问题 | 中 | 高 | 保留 JWT 回退方案 |
| 成本增加 | 高 | 低 | 监控用量、优化查询 |

| ID | 验收标准 | 测试方法 |
|----|----------|----------|
| DB-001 | 所有 D1 数据完整迁移到 Supabase | 数据校验脚本比对记录数 |
| DB-002 | 用户登录功能正常 | E2E 测试：登录 → 获取用户信息 |

### 2.2 API Gateway 设计

#### 2.2.1 业务场景

**场景 1: 统一认证**
> 所有 API 请求经过 Gateway 统一验证 JWT Token，无需每个服务单独实现。

**场景 2: 限流保护**
> 恶意用户频繁请求 AI 接口，Gateway 自动限流保护后端服务。

**场景 3: 集中监控**
> 所有 API 调用日志集中收集，支持实时监控和告警。

#### 2.2.2 技术方案

**核心功能模块**：

| 模块 | 功能 | 实现方案 |
|------|------|----------|
| 认证中间件 | JWT 验证、用户身份注入 | Hono middleware |
| 限流器 | IP/用户级别限流 | Token Bucket 算法 |
| 日志中间件 | 请求/响应日志记录 | 结构化日志 + Cloudflare Analytics |
| 错误处理 | 统一错误响应格式 | 标准化错误码映射 |
| 版本管理 | API 版本路由 | /v1/*, /v2/* 路由前缀 |

**限流策略**：

| API 类型 | 限流规则 | 说明 |
|----------|----------|------|
| 认证接口 | 10 次/分钟/IP | 防止暴力破解 |
| AI 对话 | 20 次/分钟/用户 | 保护 AI 资源 |
| 普通 CRUD | 100 次/分钟/用户 | 正常业务限制 |
| 协作接口 | 200 次/分钟/用户 | 高频操作 |

### 2.3 实时协作基础设施 Phase 1

#### 2.3.1 业务场景

**场景 1: 多人实时编辑**
> 用户 A 和用户 B 同时编辑同一个项目，A 的修改实时同步到 B 的屏幕上。

**场景 2: 在线状态显示**
> 项目成员可以看到其他协作者的在线状态（在线/离线/正在编辑）。

**场景 3: 协作锁机制**
> 用户 A 正在编辑流程图，用户 B 只能查看，无法同时编辑。

#### 2.3.2 技术方案

```
┌──────────────────────────────────────────────────────────────────┐
│                   Real-time Collaboration Architecture            │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  ┌─────────────┐         ┌─────────────┐                        │
│  │   Client A  │         │   Client B  │                        │
│  │  (WebSocket)│         │ (WebSocket) │                        │
│  └──────┬──────┘         └──────┬──────┘                        │
│         │                       │                               │
│         └───────────┬───────────┘                               │
│                     │                                           │
│         ┌───────────▼───────────┐                               │
│         │    WebSocket Server   │                               │
│         │  (Cloudflare Durable  │                               │
│         │       Objects)        │                               │
│         └───────────┬───────────┘                               │
│                     │                                           │
│    ┌────────────────┼────────────────┐                          │
│    │                │                │                          │
│    ▼                ▼                ▼                          │
│ ┌────────┐   ┌────────────┐   ┌────────────┐                   │
│ │Presence│   │Collab Lock │   │Change Sync │                   │
│ │Service │   │  Service   │   │  Service   │                   │
│ └────────┘   └────────────┘   └────────────┘                   │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**核心功能模块**：

| 模块 | 功能 | 实现方案 |
|------|------|----------|
| WebSocket 连接管理 | 连接建立/维护/断开 | Cloudflare Durable Objects |
| 在线状态服务 | 用户上线/下线/心跳 | Presence Service |
| 协作锁服务 | 资源锁定/释放/超时 | Redis-like KV 存储 |
| 变更同步服务 | 操作广播/冲突检测 | CRDT 或 OT 算法 |

**协作锁机制**：

```typescript
interface CollaborationLock {
  resourceId: string;      // 被锁定的资源 ID
  resourceType: 'project' | 'flow' | 'page';
  lockedBy: string;        // 锁持有者用户 ID
  lockedAt: Date;          // 锁定时间
  expiresAt: Date;         // 过期时间（防止死锁）
}

// 锁定流程
1. 用户请求锁定资源
2. 检查资源是否已被锁定
3. 如未锁定，创建锁并返回成功
4. 如已锁定，返回锁定者信息
5. 锁持有者释放或超时自动释放
```
