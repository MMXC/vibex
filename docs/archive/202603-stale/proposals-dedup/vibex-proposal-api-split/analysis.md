# 需求分析报告: API服务拆分

**项目**: vibex-proposal-api-split  
**阶段**: analyze-requirements  
**分析日期**: 2026-03-19  
**分析师**: Analyst Agent

---

## 1. 执行摘要

当前VibeX使用单体API架构，部分服务(如AI生成、文件处理)可能导致性能瓶颈。API服务拆分可提升系统可扩展性和性能。

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| API架构 | 单体 | 微服务化 |
| 扩展性 | 低 | 高 |
| 工作量 | - | 5-7天 |

---

## 2. 问题定义

### 2.1 核心问题

| # | 问题 | 影响 | 优先级 |
|---|------|------|--------|
| 1 | 单体API性能瓶颈 | 响应慢 | P0 |
| 2 | AI生成阻塞主服务 | 用户等待 | P1 |
| 3 | 文件处理资源占用 | 资源争抢 | P1 |

### 2.2 用户场景

**场景**: 用户提交大量需求分析
- 当前: 所有请求走同一API，可能阻塞
- 期望: AI生成异步处理，不影响主流程

---

## 3. 现状分析

### 3.1 当前架构

```
vibex-backend/
├── src/
│   ├── services/
│   │   ├── ai-generation/    # AI生成
│   │   ├── file-processing/ # 文件处理
│   │   └── .../
│   └── routes/
```

### 3.2 性能瓶颈点

| 服务 | 瓶颈 | 影响 |
|------|------|------|
| AI生成 | CPU密集 | 阻塞其他请求 |
| 文件上传 | IO密集 | 响应超时 |
| 知识库查询 | DB密集 | 慢查询 |

---

## 4. 解决方案

### 4.1 方案: API服务拆分

```
服务拆分:
├── api-gateway     # 统一入口
├── ai-service      # AI生成服务
├── file-service    # 文件处理服务
└── kb-service     # 知识库服务
```

### 4.2 技术选型

| 组件 | 推荐方案 | 理由 |
|------|----------|------|
| 网关 | Nginx/Cloudflare | 简单可靠 |
| AI服务 | Cloudflare Workers | 按需扩展 |
| 消息队列 | BullMQ | Redis兼容 |
