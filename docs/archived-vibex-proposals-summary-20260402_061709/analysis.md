## 技术方案选项

### 方案 A: 渐进式改良（推荐）

**思路**: 按依赖顺序逐个解决问题，每 1-2 周有可见交付

**实施路径**:
```
Sprint 0: D-001 + D-002 + R-P0-2 (2.5h)  ← CI基线
Sprint 1: C-P0-1 + R-P1-2 + P-P2 (11-12h) ← 用户体验
Sprint 2: A-P0 + T-P1 (10-14h)            ← 架构基础
Sprint 3: C-P0-2 + A-P0 延续 (8-12h)     ← Store完整拆分
Sprint 4: T-P0 + A-P2 (18-25h)           ← E2E + 可视化
```

**优点**: 风险分散，每阶段有交付
**缺点**: 总工期较长（5-6周）

### 方案 B: 大爆炸重构

**思路**: 一次性解决所有问题，重建基础设施

**实施路径**:
```
Phase 1: 冻结功能开发 (1周)
  - 修复TS错误
  - 重构canvasStore
  - 统一三树交互
  - 重建E2E框架
Phase 2: 功能回归验证 (1周)
Phase 3: 恢复功能开发
```

**优点**: 总工期短（2-3周）
**缺点**: 风险集中，无中间交付，团队压力大
