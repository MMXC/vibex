# Design: VibeX 全栈导出模式

Generated by /office-hours on 2026-03-25
Status: DRAFT
Mode: Startup
Founder: Alex Yu / 小羊

## Problem Statement

独立开发者做 MVP 验证时，vibe coding 跑得爽但最终成品不可控。缺少一个从"想法"到"可交付的确认产物"的受控路径。

## Target User & Narrowest Wedge

**用户**: 和 Alex 一样的独立开发者——需要自己做出可用产品，用于 MVP 验证。

**核心痛点**:
- 原型交互不够完整
- 生成没有澄清过程
- 过程不可控

**最小价值**: 独立开发者把"需求"通过 checkpoint pipeline 确认成组件节点 JSON，再导出为可直接开发的项目。

## Core Pipeline

```
需求录入 → 流程节点（可勾选/编辑/保存）→ 组件节点（可勾选/编辑/保存）→ 原型页面 → 导出
```

**Checkpoint 机制**: 每步生成后，人类确认 JSON，AI 基于确认结果继续下一步。不允许跳步。

## Premises

1. **VibeX 的核心价值** = vibe coding 和生产代码之间的**桥梁**——让独立开发者能"验证后再交付"
2. **竞争优势** = 不是 AI 生成本身（谁都能接 API），而是**人类 checkpoint 控制**——每一步都是人确认过的 JSON，不是 AI 闷头吐的代码
3. **最小闭环** = 独立开发者把"需求"变成"可确认的组件节点 JSON"，再导出给开发——这一段就够了

## Chosen Approach: 全栈导出模式

### 导出产物

- Next.js/React 项目结构
- TypeScript 类型定义（从组件节点 JSON 生成）
- API mock 文件（基于流程节点生成）
- 路由结构
- 可直接在 Cursor/VSCode 打开开发

### 与竞品差异

- 不是 AI 生代码（Cursor/GitHub Copilot 已经在做）
- 是**人类控制 + AI 生成 = 可信赖的产出物**

### MVP 验证目标

导出 Next.js 单页项目，能跑，有类型，可直接改。

## Success Criteria

1. Alex 本人能导出第一个可用项目
2. 导出的代码可直接在 Cursor 打开开发
3. 流程节点/组件节点可勾选、编辑、保存

## Open Questions

- 导出项目格式：Next.js App Router 还是 Pages Router？
- 状态管理：要不要内置？
- API mock 格式：MSW 还是直接 JSON 文件？

## What I Noticed

- 你很清楚自己要解决什么问题，没有"先做着看"的模糊感
- 回答具体、直接，模块化可控的描述非常清晰
- 第一用户驱动——不是调研出来的需求，是自己真实在用的痛点
