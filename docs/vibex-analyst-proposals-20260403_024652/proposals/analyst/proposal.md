## 2. 提案详情

### A-1: 画布 Phase 状态引导层

**分析视角**: 用户体验 + 信息架构

**问题**: 用户进入 editor 后不知道自己在哪个 Phase，三树切换时无视觉锚点，新用户缺乏引导。

**建议方案**: Phase 指示器 + 引导卡片（2-3h）

**验收标准**:
- Phase 指示器在画布顶部始终可见
- 切换 Phase 时指示器状态同步更新
- 首次进入画布显示引导卡片（仅一次）

---

### A-2: 设计产物分析指标体系

**分析视角**: 产品分析 + 数据驱动

**问题**: 无使用数据，无法回答哪些功能被真正使用、用户在哪步停留最久。

**建议方案**: 轻量埋点 + 分析仪表盘（4-6h）

**验收标准**:
- `/analytics` 页面可访问
- 显示最近 7 天项目完成率
- 无个人用户级别追踪

---

### A-3: 协作分享与版本快照

**分析视角**: 用户旅程 + 价值实现

**问题**: 完成 DDD 建模后无协作分享和版本对比能力，产出物无人消费。

**建议方案**: 协作分享 + 版本快照（8-12h）

**验收标准**:
- 项目可生成只读分享链接
- 协作者可通过邮箱接受邀请
- 设计版本快照支持命名和保存

---

### A-4: 端内 Feedback 收集机制

**分析视角**: 用户声音 + 质量闭环

**问题**: 用户遇到问题只能通过外部渠道反馈，路径不清晰，PM 难以归类。

**建议方案**: Floating Feedback 按钮 + 分类表单（3-4h）

**验收标准**:
- 反馈按钮在画布页面始终可见
- 提交后显示确认 Toast
- PM 频道收到通知
- 反馈自动带上当前 URL 和 Phase

---

### A-5: CI 质量仪表盘与趋势追踪

**分析视角**: 质量工程 + 持续改进

**问题**: 每次迭代质量依赖人工验收，无法量化"这次发布比上次稳定吗"。

**建议方案**: CI 质量仪表盘 + 趋势图（3-4h）

**验收标准**:
- `/quality` 页面显示 E2E 通过率趋势
- 显示 TypeScript 错误数趋势
- 通过率 < 90% 时 Slack 报警
