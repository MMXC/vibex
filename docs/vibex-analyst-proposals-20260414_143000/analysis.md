## 1. 业务场景分析

### 业务价值

Analyst 角色提案聚焦于 VibeX 项目的**可行性评估与风险治理**，目标是在任何功能/架构变更进入实施前完成独立评审，防止团队陷入"做了才发现不可行"的困境。

核心业务价值：
- **降低返工成本**：通过前置风险识别，减少实施阶段的返工
- **辅助 Coord 决策**：提供结构化的可行性评估，让决策有据可依
- **构建分析基础设施**：建立跨项目可复用的分析框架和风险矩阵模板

### 目标用户

| 用户 | 使用场景 |
|------|---------|
| Coord Agent | 读取 Analyst 评审结论，支撑 Sprint 决策 |
| Dev Agent | 参考可行性评估中的技术风险项，规避实施陷阱 |
| PM Agent | 对照验收标准，确认需求是否足够清晰 |
| Architect Agent | 参考技术可行性评估，验证架构方案是否被正确理解 |

## 2. 核心 JTBD（Jobs-To-Be-Done）

1. **When** 一个新提案到达，**I want** 在 24h 内获得结构化可行性评估，**So that** 不会因信息不足而做错决策
2. **When** 多个提案同时需要排序，**I want** 看到统一的风险矩阵对比，**So that** 能基于风险/收益做出最优优先级决策
3. **When** Architect 方案存在不确定性，**I want** 有独立的技术评审视角，**So that** 不是 Architect 自己审查自己
4. **When** Sprint 进行中出现预期外阻塞，**I want** 追溯到对应的风险预警记录，**So that** 能快速判断是否需要 escalation
5. **When** 项目复盘时，**I want** 对照原始可行性评估与实际结果，**So that** 持续改进分析质量

## 3. 技术方案选项

### 方案A：文档驱动分析（当前采用）

**描述**: Analyst 产出 `docs/analysis/<project>-feasibility.md`，包含风险矩阵、工期估算、验收标准。  
**优势**:
- 低门槛，任何成员都能读取
- 版本化：Git 追踪历史
- 可被 subagent 批量处理

**劣势**:
- 分散在多个目录，检索需要 grep
- 无法自动聚合统计风险分布

### 方案B：结构化数据库 + 文档双轨

**描述**: 在 D1 数据库中维护风险条目结构化数据，同时生成 Markdown 文档。  
**优势**:
- 支持按风险等级/项目/日期聚合查询
- 可接入 Dashboard 可视化

**劣势**:
- 初期工程成本高（需要 schema 设计 + API）
- 团队成员无法直接编辑

**当前决策**: 方案A，待规模扩大后评估方案B。

## 5. 初步风险识别

### 技术风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 提案信息不完整，无法评估 | 🟠 中 | 建立"阻塞触发器"：信息不足时立即反馈 |
| 评估准确性依赖已知知识边界 | 🟡 低 | 对未知领域明确标注"不确定度高" |

### 业务风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 分析结论被跳过直接进实施 | 🔴 高 | Coord 流程中加入 Analyst 评审 gate |
| 分析过于保守导致好提案被驳回 | 🟠 中 | 验收标准明确，让结论可被质疑和推翻 |

### 依赖风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| PM 提案时间线压缩 Analyst 窗口 | 🟠 中 | 明确 Analyst 评审 SLA（24h） |
| Architect 评审与 Analyst 评审产生冲突 | 🟡 低 | 以 Coord 最终裁决为准 |

## 6. 验收标准

- [ ] 每个提案到达后 24h 内产出 feasibility.md
- [ ] 每份分析包含风险矩阵（技术/业务/依赖三维）
- [ ] 工期估算误差 < 30%（sprint 结束后回验）
- [ ] 每份分析包含明确的结论（推荐/不推荐/有条件推荐）
- [ ] 提案包含 `## 执行决策` 段落，状态字段填写完整
- [ ] 被 Analyst 标记"高风险"的提案，实施阶段实际风险发生率 < 20%
