# Spec: E2 - 工时估算标准和回验格式

## E2.1 工时估算格式

```markdown
| 工时项 | 估算值 | 乐观 | 悲观 | 估算依据 |
|--------|--------|------|------|---------|
| [阶段/任务名] | Xh | X-25% | X+50% | [依据说明] |
| **合计** | **Yh** | **Y-25%** | **Y+50%** | |
```

### 估算规则

| 场景 | 估算原则 |
|------|---------|
| 熟悉的领域 + 有直接参考 | 估算值 ± 10% |
| 熟悉的领域 + 无直接参考 | 估算值 + 缓冲 25% |
| 陌生领域 + 无直接参考 | 乐观 = 估算值 × 1.5，悲观 = 估算值 × 2.5 |
| 涉及外部依赖（API/设计稿） | 额外 + 50% 缓冲 |
| 涉及数据迁移或持久化变更 | 额外 + 100% 缓冲 |

## E2.2 估算依据必须包含

1. **参考历史**：基于哪个已完成项目的工时数据
2. **类比基准**：与哪个已有功能类似，规模比例是多少
3. **不确定性来源**：哪些因素可能导致偏差

**有效示例**: 
```
"基于 canvas-button-cleanup 项目（完成，8h），本提案复杂度约为其 1.5 倍，
因此估算 12h。不确定性来源：React Flow 重构可能引入隐藏耦合。"
```

**无效示例**: "估算 12h。"

## E2.3 回验格式

```markdown
回验记录（完成后填写）:
- 实际工时: 14h
- 偏差率: (14-12)/12 = 16.7%
- 偏差原因: [说明]
- 改进建议: [下次估算如何改进]
```

### 偏差率计算

```
偏差率 = abs(实际工时 - 估算值) / 估算值
```

### 偏差率分级

| 偏差率 | 评级 | 处理 |
|--------|------|------|
| < 15% | 🟢 准确 | 记录为基准 |
| 15-30% | 🟡 可接受 | 分析原因 |
| > 30% | 🔴 偏差大 | 复盘，改进估算方法 |

## E2.4 Sprint 回验追踪

在项目完成后，由 Coord 触发回验：

```markdown
## 工时回验追踪

| 提案 ID | 估算值 | 实际值 | 偏差率 | 评级 |
|---------|--------|--------|--------|------|
| proposal-20260410-001 | 8h | 10h | 25% | 🟡 |
| proposal-20260411-002 | 4h | 4h | 0% | 🟢 |
```

每月汇总：
- 平均偏差率
- 最大偏差案例
- 估算方法改进建议
