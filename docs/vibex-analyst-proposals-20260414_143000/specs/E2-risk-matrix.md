# Spec: E2 - 风险矩阵格式和等级定义

## E2.1 风险等级定义

| 等级 | 定义 | 触发条件 |
|------|------|---------|
| 🔴 高 | 严重影响核心功能或导致项目无法推进 | 阻塞用户完成核心操作 / 技术方案不可行 / 安全/合规问题 |
| 🟠 中 | 影响开发效率或交付时间，但有缓解路径 | 依赖外部资源 / 跨团队协作 / 复杂实现 |
| 🟡 低 | 影响用户体验，但不影响核心功能 | UI 细节 / 性能优化 / 边缘情况 |

## E2.2 风险矩阵格式

```markdown
| 风险 | 等级 | 缓解措施 | 残余风险 |
|------|------|---------|---------|
| [描述，如：API 响应超时导致 UI 僵死] | 🔴 高 | [具体措施，如：添加 5s 超时 + 重试 1 次 + 降级文案] | [残余风险，如：超时后用户需手动重试] |
```

## E2.3 风险分类

每个分析必须包含以下三类风险（至少各 1 个）：

| 类别 | 定义 | 示例 |
|------|------|------|
| **技术风险** | 技术方案、实现难度、依赖组件 | API 不可用 / 性能不达标 / 技术栈不兼容 |
| **业务风险** | 业务价值、用户影响、优先级 | 需求变更 / 用户不接受 / ROI 为负 |
| **依赖风险** | 上游/下游依赖、时间窗口 | 后端未就绪 / UI 依赖设计稿 / Sprint 截止临近 |

## E2.4 缓解措施质量标准

有效的缓解措施必须包含：
1. **具体行动**：不是"注意"或"待定"，而是具体的操作
2. **责任人**：谁负责执行缓解
3. **时间点**：何时完成缓解
4. **验证方式**：如何确认缓解生效

**无效示例**: "风险：API 超时。缓解：添加重试机制。"
**有效示例**: "风险：API 超时（>5s）。缓解：在 apiClient 中添加 AbortController，5s 超时自动 abort，重试 1 次；验证方式：测试覆盖率 > 80%；责任人：Dev Agent；完成时间：E2 结束时。"

## E2.5 残余风险评估

每个风险必须有残余风险评估：
- **完全可接受**：缓解措施完全消除了风险影响
- **部分残余**：缓解降低了影响但未完全消除
- **不可接受**：即使缓解也无法接受，应考虑放弃提案
