# Spec: E3 - 评审 Gate 决策标准和采纳追踪规格

## E3.1 三种评审结论的定义

### 推荐 ✅

**触发条件（必须全部满足）**:
1. 技术可行性 = 高（技术方案成熟，有直接参考）
2. 业务价值 > 0（对用户或产品有可量化收益）
3. 工时估算合理（偏差率预期 < 30%）
4. 风险可接受（无 🔴 高风险，或有明确缓解措施）

**输出格式**:
```markdown
**结论**: 推荐

理由：[简洁陈述为什么值得做，引用数据支撑]
Sprint 建议：[建议的 Sprint 编号或优先级排序]
```

---

### 不推荐 ❌

**触发条件（满足任一）**:
1. 技术不可行（没有可行的技术路径，或依赖不存在的组件）
2. 业务价值 = 0（对用户无收益，或与产品方向冲突）
3. 风险无法缓解（🔴 高风险且无缓解路径）
4. 工时估算无限大（> 2 Sprint 工作量且无法拆分）

**输出格式**:
```markdown
**结论**: 不推荐

驳回原因：
1. [具体原因1，不能模糊]
2. [具体原因2，不能模糊]

如果条件变化，可重新评估的条件：
- [条件1]
- [条件2]
```

**⚠️ 禁止使用模糊驳回**:
- ❌ "综合考虑不建议" → 必须列出具体的技术/业务/风险原因
- ❌ "风险较高" → 必须说明是什么风险，缓解措施是什么
- ❌ "工作量较大" → 必须给出具体估算值

---

### 有条件推荐 ⚠️

**触发条件（满足所有）**:
1. 技术可行（有可行路径，但有未完全解决的风险）
2. 业务价值 > 0
3. 存在可识别但可管理的风险（必须列出具体条件）

**输出格式**:
```markdown
**结论**: 有条件推荐

必须满足的条件：
1. [条件1，如：后端 API 在 Sprint 开始前就绪]
2. [条件2，如：设计稿在 3 天内交付]

若条件无法满足，降级为：[不推荐 / 延期]
```

---

## E3.2 Coord 采纳追踪

### Coord 决策记录格式

在 analysis.md 末尾添加：

```markdown
---

## Coord 采纳记录

- **最终决策**: [通过 / 驳回 / 延期]
- **采纳时间**: YYYY-MM-DD HH:MM
- **决策理由**: [具体理由，若推翻 Analyst 结论必须说明原因]
- **与 Analyst 结论一致**: [是 / 否]
- **若不一致，原因**: [Coord 与 Analyst 意见不同的原因]
```

### 一致性统计

每月由 Coord 生成：

```markdown
## Analyst-Coord 一致性月报（YYYY-MM）

| 提案 ID | Analyst 结论 | Coord 决策 | 一致 |
|---------|-------------|-----------|------|
| [ID] | 推荐 | 通过 | ✅ |
| [ID] | 不推荐 | 驳回 | ✅ |
| [ID] | 推荐 | 驳回 | ❌ |

**采纳率**: X/Y = Z%
**误判案例**: [列表，若有]
```

---

## E3.3 误判复盘流程

当 Coord 推翻 Analyst 结论时：

1. **记录**: 在 analysis.md 的 Coord 采纳记录中填写原因
2. **复盘**: 提取关键教训，写入 `docs/learnings/` 目录
3. **改进**: 更新模板或判断标准，防止同类误判

```markdown
# Learnings: Analyst 误判复盘

**日期**: YYYY-MM-DD
**提案 ID**: proposal-YYYYMMDD-NNN
**误判类型**: [高风险未识别 / 工时严重低估 / 业务价值误判]

**发生了什么**: [客观描述]

**为什么 Analyst 误判**: [根本原因分析]

**如何避免**: [具体改进措施]

**模板/标准更新**: [是否需要更新模板或标准]
```
