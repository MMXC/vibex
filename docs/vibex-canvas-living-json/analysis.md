# Analysis: VibeX Canvas 范式重构 — 选择驱动的 Living JSON

**项目**: vibex-canvas-living-json
**日期**: 2026-03-31
**状态**: Draft v2（已根据用户澄清更新）

---

## 1. 业务场景分析

### 1.1 范式定义

**核心理念**：画布 = 选择驱动的可视化 JSON 编辑器

| 维度 | 说明 |
|------|------|
| 初始状态 | 三棵空树（JSON 结构存在，节点数组为空）|
| 入口 | 对话框输入原始需求 → AI 生成 context 树 |
| 编辑方式 | 选择卡片 + 点击生成按钮，不存在自由文本编辑 |
| 联动规则 | 修改上游不自动联动下游，用户手动触发生成命令 |
| 重置 | 清空 JSON 回到三棵空树 |

### 1.2 生成命令体系（MVP）

| 命令 | 触发条件 | 产出 |
|------|---------|------|
| 原始需求生成 | 对话框输入需求，未选中任何卡片 | 生成 context 树 |
| 生成业务流程 | 选中 context 卡片 + 点击按钮 | 生成 flow 树 |
| 重新生成上下文 | 选中 context 卡片 + 点击按钮 | 重新生成 context 树（覆盖）|
| 重新生成流程 | 选中 flow 卡片 + 点击按钮 | 重新生成 flow 树（覆盖）|
| 生成组件树 | 选中 flow 卡片 + 点击按钮 | 生成 component 树 |

### 1.3 目标用户

- **产品/架构**：通过对话框输入需求，快速生成 DDD 模型
- **团队**：画布作为 PRD 可视化原型，导出 JSON 后交给开发

---

## 2. 技术方案

### 方案 A：选择驱动生成（推荐）

用户选择节点 → 点击命令按钮 → AI 根据选中内容生成

**优点**：交互明确，用户知道每次触发了什么
**缺点**：比完全自由编辑约束更多

### 方案 B：AI 自主推断

用户编辑任意位置 → AI 自动推断应该更新哪棵树

**优点**：更流畅
**缺点**：用户可能不理解为什么 AI 动了不该动的树

---

## 3. 关键设计决策（已确认）

| 决策 | 结论 |
|------|------|
| 初始状态 | 三棵空树 |
| phase 导航 | 完全移除 |
| 确认门控 | 完全移除 |
| 上游变更联动 | 不联动，用户手动触发 |
| 命令对话框 | MVP 包含：原始需求、生成下一步、重新生成 |
| confirmed 字段 | 移除，替换为选中状态 |

## 4. 待确认项

1. **选中多张卡片时**：生成是针对每张卡单独生成，还是基于选中集合批量生成？
2. **生成覆盖策略**：重新生成是"追加"还是"替换"选中节点的产出？
3. **对话框输入历史**：是否需要保存历史记录？
