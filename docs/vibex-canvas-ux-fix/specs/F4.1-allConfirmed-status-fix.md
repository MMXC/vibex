# F4.1 — allConfirmed 检查 status === 'confirmed'

**文件**: `vibex-fronted/src/components/canvas/BoundedContextTree.tsx`
**行号**: 463
**优先级**: Medium

---

## 当前有 Bug 的代码

```typescript
const allConfirmed = contextNodes.length > 0 && contextNodes.every((n) => n.isActive !== false);
```

**Bug**:
- `allConfirmed` 检查 `isActive !== false`
- 但 checkbox（`ContextCard.tsx` 行 237）操作设置的是 `status === 'confirmed'`
- `isActive` 和 `status` 是两个独立字段：
  - `isActive`: 由 AI 生成/cascade 流程设置，可能被其他操作单独修改
  - `status`: 由用户 checkbox 操作（`toggleContextNode`）设置
- 两者可能不同步：节点可显示"已确认"（checkbox 勾选）但 `allConfirmed` 仍为 `false`

---

## 修复方案

```typescript
const allConfirmed = contextNodes.length > 0 && contextNodes.every((n) => n.status === 'confirmed');
```

**改动点**:
- 将 `n.isActive !== false` 改为 `n.status === 'confirmed'`
- 使 `allConfirmed` 与 checkbox 勾选状态（`checked={node.status === 'confirmed'}`）保持一致

---

## 关联修复（F4.2）

此修复依赖 F4.2（`handleConfirmAll` 必须真正设置 `status === 'confirmed'`）。若 F4.2 未实施，`allConfirmed` 改为检查 `status` 后，由于 `handleConfirmAll` 不设置 `status`，`allConfirmed` 永远为 `false`，面板永远锁定。

**必须与 F4.2 配对实施。**

---

## Panel lock 一致性检查（F4.3）

修复 `allConfirmed` 后，需要确认 `BusinessFlowTree.tsx` 的 `inactivePanel` 也读取同一标志：

```typescript
// BusinessFlowTree.tsx — 当前代码（行 ~843）
if (!isActive) {
  return (
    <div className={styles.inactivePanel}>
      <p>请先完成上下文树后解锁</p>
    </div>
  );
}
```

父组件（`BoundedContextTree` 或 `CanvasPage`）需要将 `allConfirmed` 结果透传给 `BusinessFlowTree` 的 `isActive` prop，确保两处状态一致。

| T5 | 空数组 | `[]` | `false` |
| T6 | 单个节点 `status === 'confirmed'` | 1个 confirmed | `true` |
