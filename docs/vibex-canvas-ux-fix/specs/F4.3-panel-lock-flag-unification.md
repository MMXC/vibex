# Spec: F4.3 — Panel Lock 与 allConfirmed 标志统一

**Story ID**: F4.3
**所属 Epic**: E4
**文件**: `vibex-frontend/src/components/canvas/BusinessFlowTree.tsx`
**改动位置**: `inactivePanel` 锁定逻辑

---

## 现状（Bug）

面板锁定机制读取 `isActive` prop：

```typescript
// BusinessFlowTree.tsx — panel lock
if (!isActive) {
  return <div className={styles.inactivePanel}><p>请先完成上下文树后解锁</p></div>;
}
```

`allConfirmed`（修复后）读取 `status === 'confirmed'`，两者依赖不同字段，可能不同步。

---

## 改动规格

### 改动点：确认 panel lock 与 allConfirmed 读取同一标志

**分析结论**：修复 F4.1 后，`allConfirmed` 改为 `contextNodes.every(n => n.status === 'confirmed')`。

**方案选择**：

选项 A（推荐）：Panel lock 也读 `status === 'confirmed'`
```typescript
const allConfirmed = contextNodes.length > 0 && contextNodes.every(n => n.status === 'confirmed');
// ...
if (!allConfirmed) {
  return <div className={styles.inactivePanel}><p>请先完成上下文树后解锁</p></div>;
}
```

选项 B：保持 panel lock 读 `isActive`，统一 `toggleContextNode` 同时设置双字段（已在 F4.2 中覆盖）

**验证步骤**：

1. 扫描 `BusinessFlowTree.tsx` 中所有读取 `isActive` 的位置
2. 对每个位置，判断是否应改为读 `status === 'confirmed'`
3. 确保父组件透传的 prop 与子组件内部判断一致

### 审计清单

- [ ] `inactivePanel` 的锁定条件已与 `allConfirmed` 结果一致
- [ ] `BusinessFlowTree` 内无其他位置使用 `isActive !== false` 判断"上下文树完成"状态
- [ ] `advancePhase()` 触发后，panel lock 状态同步更新（无状态延迟）

---

## 测试场景

| # | 场景 | 前置条件 | 操作 | 预期结果 |
|---|------|---------|------|---------|
| T1 | allConfirmed true → 面板解锁 | 所有节点 status === 'confirmed' | 渲染 | inactivePanel 不出现，正常内容出现 |
| T2 | allConfirmed false → 面板锁定 | 部分或全部节点 status !== 'confirmed' | 渲染 | inactivePanel 出现，文案"请先完成上下文树后解锁" |
| T3 | 状态切换 | 点击确认所有节点 | 点击后 | 面板从锁定变为解锁（200ms 内） |
