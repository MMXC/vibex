## 1. 业务场景分析

### 要解决的核心问题

**本质**：将 VibeX 从"一次性需求→原型生成器"扩展为"软件工程全流程文档化管理平台"。

现有流程是线性顺序的（需求→上下文→流程→组件），用户完成一个阶段就结束。本需求想把所有工程文档章节整合到一个交互式画布里，核心价值是：
- **结构化沉淀**：工程文档（需求/上下文/流程/数据/架构）不再是离散文件，而是结构化卡片
- **可编辑关系**：章节之间、卡片之间的关系可可视化编辑
- **AI 驱动生成**：各章节均可通过 AI 对话生成

### 目标用户

| 用户 | 场景 |
|------|------|
| 产品经理 | 维护需求文档，直接在画布上编辑用户故事 |
| 架构师 | 可视化限界上下文和依赖关系 |
| 全栈工程师 | 查看/编辑数据模型、接口设计 |

### 核心 JTBD

| # | JTBD | 优先级 |
|---|------|--------|
| JTBD-1 | 查看和管理一个项目的所有工程文档章节（需求/上下文/流程/数据/架构） | 🔴 P0 |

| JTBD-3 | 可视化编辑卡片之间的关系（树/DAG） | 🟠 P1 |

## 2. 技术方案选项

### 方案 A（推荐 MVP）：复用现有 Canvas 基础设施，横向扩展

**思路**：现有 `/app/canvas/CanvasPage.tsx` 已经支持三树。扩展为横向多面板，每个面板是一个章节的 DAG 视图。共享同一套 canvasStore 和 React Flow 渲染引擎。

**实现路径**：
- 新建 `/app/dds-canvas/page.tsx`（DDS = Detailed Design Specification）
- 将现有"三树"作为前 3 个章节面板
- 后续章节（数据模型/架构/接口/非功能需求）作为新面板增量加入
- 面板横向滑动用 CSS scroll-snap 或 custom scroll 方案
- AI 对话区复用现有 `ChatDrawer` 或新建 `DDSAIChat` 组件

**优点**：
- React Flow 渲染引擎、canvasStore、数据模型均可复用
- 与现有原型画布（prototype canvas）天然分离，用户心智清晰
- 增量开发，每章独立可测试

**缺点**：
- 现有 canvasStore 是否有足够抽象度支撑多章节尚需验证（代码审查发现 store 326-406 行，偏重）
- 横向"奏折"布局与现有垂直三树布局交互模式差异大

### 方案 B：从零设计独立子系统

**思路**：新建完全独立的 DDS 子系统，包含自己的 store、数据模型、渲染引擎。

**优点**：无历史包袱，可按最优设计实现。
**缺点**：过度工程。canvas 的 React Flow + Zustand 模式已验证，重写是浪费。

## 3. 可行性评估

### 技术可行性

| 需求 | 可行性 | 难度 | 说明 |
|------|--------|------|------|
| 奏折式横向布局 | ✅ 高 | 低 | CSS scroll-snap 成熟方案 |
| React Flow DAG 渲染 | ✅ 高 | 低 | 已有基础设施 |
| 多章节横向滑动切换 | ⚠️ 中 | 中 | 需处理面板状态同步和 URL 同步 |
| AI 对话生成卡片 | ✅ 高 | 中 | 复用现有 AI chat 模式 |
| 卡片 JSON 数据结构 | ✅ 高 | 中 | schema 设计是关键 |
| 关系可编辑（树/DAG） | ✅ 高 | 中 | React Flow 支持，但需 UI |
| PRD 预览汇总 | ✅ 高 | 低 | 已有 prototype preview 模式可复用 |

**结论**：技术可行，无未验证的技术风险。主要难度在工程量而非技术。

### 范围可行性

⚠️ **当前需求范围存在严重模糊，必须澄清**：

1. **章节定义不明**："需求分析/上下文分析/领域流程/数据结构/架构设计/接口设计/非功能需求/PRD预览"——这 8 个章节是全部都要还是示例？"等其他可 JSON 化章节"是什么意思？实际可能有多少章节？
2. **MVP 定义缺失**：第一期做哪几个章节？没有 MVP 定义意味着没有出口标准。
3. **卡片 JSON 字段不明**：每种卡片的字段是什么？没有 schema 设计，无法评估工作量。
4. **"AI 先生成文本，确认后产出卡片"的具体交互**：AI 输出文本后，用户怎么"确认"？手动复制粘贴？还是 AI 直接解析后填充？这两者是本质不同的交互。

## 4. 初步风险识别

### 🔴 高风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 范围无边界（"其他可 JSON 化章节"无上限） | 工期失控 | **第一期必须明确固定 3-5 个章节**，不允许增量扩展 |
| 卡片 JSON schema 设计不当导致后续重构 | 严重返工 | **先做 schema 设计评审，再开始开发** |
| 与现有 canvas 的数据共享/同步策略不明 | 数据一致性 | 定义 DDS canvas 和 prototype canvas 的数据边界 |

### 🟠 中风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 横向"奏折"交互与用户习惯不符（用户熟悉垂直，不熟悉横向） | 用户体验 | **先做交互原型（5-10 分钟 Figma）给 PM 确认，再开发** |
| AI 生成卡片内容的质量不稳定 | 用户信任 | 第一期先做"用户写卡片，AI 帮改"而非"AI 直接生成" |
| canvasStore 扩展到多章节后状态管理复杂度指数上升 | 维护成本 | 若 store 超过 500 行，拆分而非继续扩展 |

### 🟡 低风险

| 风险 | 影响 | 处理 |
|------|------|------|
| 全屏横向推开与浏览器滚动条冲突 | UI | 全屏模式禁用 body scroll |
