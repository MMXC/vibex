## 2. 技术方案选项（≥2 方案）

### 2.1 P001: 需求智能补全

#### 方案 A: 前端实时检测 + 后端澄清 API（推荐）

**实现方式**:
- 前端：监听输入事件，检测关键词（实体名、动词），发送 `/api/v1/analyze/partial`
- 后端：轻量 LLM 调用（或关键词规则）生成澄清问题
- 前端：渲染澄清追问卡片，支持用户回答，继续补充

**优点**:
- 实时交互，用户体验流畅
- 可配置澄清规则，无需每次调用 LLM
- 与现有 AI 分析 pipeline 可复用

**缺点**:
- 部分 LLM 调用有延迟
- 澄清规则需维护

**预计工时**: 4h

---

#### 方案 B: 纯后端多轮对话

**实现方式**:
- 用户输入作为初始消息
- 后端 LLM 判断是否需要澄清，追加系统消息引导
- 前端展示多轮对话流

**优点**:
- LLM 智能判断澄清时机
- 可深度追问

**缺点**:
- 每轮澄清都有 LLM 成本
- 前端改动大（多轮对话 UI）
- 延迟高（需等 LLM 回复）

**预计工时**: 7h

**推荐**: 方案 A（4h，快速上线，交互体验好）

---

### 2.2 P002: 项目搜索与过滤

#### 方案 A: 前端过滤（推荐）

**实现方式**:
- 现有 API 返回全量项目列表（通常 < 100 个）
- 前端使用 `filter()` + `debounce` 实现搜索
- 按名称/时间/状态客户端过滤

**优点**:
- 实现简单，无需后端改动
- 响应极快（< 50ms）
- 无额外 API 成本

**缺点**:
- 不适合超多项目（> 1000 个）场景
- 全量数据加载略浪费带宽

**预计工时**: 2h

---

#### 方案 B: 后端搜索 API

**实现方式**:
- 新增 `GET /api/v1/projects/search?q=&filter=`
- 后端支持数据库全文检索（Prisma full-text 或 Algolia）
- 分页返回结果

**优点**:
- 支持超大规模数据
- 可扩展排序算法（相关性/活跃度）

**缺点**:
- 需要数据库改动
- 实现复杂度高

**预计工时**: 5h

**推荐**: 方案 A（2h，先满足当前需求，后续可演进到方案 B）

---

### 2.3 P003: 团队协作空间 UI

#### 方案 A: 基于现有 KV 扩展（推荐）

**实现方式**:
- 复用 `CollaborationService`（已部署的 KV service）
- 新增团队管理 API：`/api/v1/teams`（create/list/invite）
- 前端新增团队设置页面和权限管理 UI
- 协作视图：WebSocket 推送在线成员状态

**优点**:
- KV 后端已就绪，投入少
- 与现有 KV 基础设施一致
- 支持实时协作

**缺点**:
- KV 不支持复杂查询（如成员搜索）
- 权限粒度有限

**预计工时**: 6h

---

#### 方案 B: 数据库驱动协作

**实现方式**:
- 创建 `teams`, `team_members`, `permissions` 数据库表
- 关系型数据库支持复杂权限查询
- WebSocket 推送协作事件

**优点**:
- 支持细粒度权限控制
- 可查询成员活跃度等数据

**缺点**:
- 需要数据迁移
- 后端改动量大
- 与现有 KV 重复建设

**预计工时**: 12h

**推荐**: 方案 A（6h，KV 已就绪，快速交付）

## 6. 快速产品视角扫描（5分钟）

### 用户体验问题
1. **输入质量无保障** — AI 直接生成，用户无中途干预机会，返工率高
2. **多项目场景体验差** — 10+ 项目后无法快速找到目标，只能靠记忆
3. **Tree 按钮样式割裂** — 视觉不一致影响专业感和信任度

### 功能缺失
1. **智能补全** — 最直接影响生成质量的功能，20260410 已计划但未开发
2. **团队协作 UI** — KV 后端空转，前端为零
3. **版本对比** — 无法追溯变更，企业场景硬伤
4. **快捷键** — 效率用户强烈需求

### 流程断点
1. **需求输入 → AI 分析** — 无澄清机制，输入模糊时直接生成错误结果
2. **生成组件 → 归属 Flow** — flowId 关联无 E2E 验证，数据错乱风险
3. **项目管理 → 协作** — KV 就绪但 UI 缺失，流程断裂
