## 1. 业务场景分析

### 1.1 核心产品定位

VibeX 是一个 AI 驱动的 DDD（领域驱动设计）产品建模平台。用户通过对话式需求分析，AI 生成领域模型 + 可视化流程 + 原型页面。

**目标用户**：
| 用户类型 | 特征 | 当前痛点 |
|---------|------|---------|
| 产品经理 | 熟悉业务，不懂技术 | 难以清晰表达系统边界 |
| 开发工程师 | 懂技术，业务理解浅 | 领域模型设计困难 |
| 架构师 | 需要快速验证想法 | 缺乏高效工具 |
| 初创团队 | 资源有限 | 没有专人做设计 |

### 1.2 用户旅程分析

```
用户旅程阶段          当前状态           问题识别
─────────────────────────────────────────────────────
首次访问              → 落地页           🔴 无引导
输入需求              → 空白输入框       🔴 门槛高
AI 分析               → 生成结果         🟡 质量不稳
查看结果              → 领域模型         🟢 正常
创建项目              → 项目列表         🟡 无搜索
协作                  → 不可用           🔴 功能缺失
```

### 1.3 核心问题归纳

| 问题类别 | 影响 | 紧迫度 |
|---------|------|--------|
| 新用户引导缺失 | 首次使用流失率高 | P0 |
| 需求模板库缺失 | 输入质量低，AI 生成差 | P0 |
| 项目管理功能弱 | 多项目场景无法使用 | P1 |
| 协作功能缺失 | 企业场景无法使用 | P1 |
| 版本管理缺失 | 无法追溯变更 | P1 |
| 快捷键/离线等 | 高级用户效率低 | P2 |

## 2. 技术方案选项

### 2.1 P001 需求模板库

#### 方案 A: 本地 Markdown 模板（推荐）

**实现方式**:
- 在 `/content/templates/` 目录存放 Markdown 模板文件
- 模板结构：`title`, `industry`, `entities[]`, `contexts[]`, `example_requirements`
- 前端直接读取文件系统

**优点**:
- 实现简单，无后端依赖
- 部署即生效，无需数据库改动
- 内容易于维护和扩展

**缺点**:
- 无法动态更新，需要重新部署
- 缺乏用户贡献模板机制

**预计工时**: 3h

---

#### 方案 B: 数据库 + API 模板

**实现方式**:
- 创建 `Template` 数据库表
- 提供 CRUD API：`/api/v1/templates`
- 支持用户提交自定义模板

**优点**:
- 支持动态更新
- 支持用户贡献
- 便于数据分析

**缺点**:
- 增加后端工作量
- 需要数据迁移
- 复杂度更高

**预计工时**: 6h

**推荐**: 方案 A（快速上线，方案 B 作为后续演进）

---

### 2.2 P002 新手引导流程

#### 方案 A: Overlay + Tooltip 引导（推荐）

**实现方式**:
- 复用现有 `<CanvasOnboardingOverlay />`
- 新增 `OnboardingTour` 组件，支持步骤配置
- 使用 `@reactour` 或自定义实现

**优点**:
- 实现成熟，用户熟悉
- 可复用现有 overlay
- 支持 skip/skip-all

**缺点**:
- 需要处理多页面引导
- 样式一致性需要维护

**预计工时**: 3h

---

#### 方案 B: 引导页独立路由

**实现方式**:
- 创建 `/onboarding` 路由
- 全屏引导体验
- 完成后跳转首页

**优点**:
- 体验一致性好
- 不影响现有页面

**缺点**:
- 需要处理路由跳转
- 引导中途离开体验差
- 开发量大

**预计工时**: 5h

**推荐**: 方案 A（基于现有 overlay 扩展，快速）

---

### 2.3 P005 团队协作空间

#### 方案 A: KV 驱动轻量协作

**实现方式**:
- 利用已有 Cloudflare Workers KV（COLLABORATION_KV）
- 扩展现有 CollaborationService
- 前端增加团队管理 UI

**优点**:
- 复用现有基础设施
- 并发安全已验证
- 部署简单

**缺点**:
- KV 容量有限（大型团队场景）
- 不适合复杂权限场景

**预计工时**: 6h

---

#### 方案 B: Prisma 数据库协作

**实现方式**:
- 创建 `Team`, `TeamMember`, `ProjectPermission` 表
- 关系型数据，支持复杂查询
- 提供 REST API

**优点**:
- 数据完整性强
- 支持复杂权限
- 可扩展性好

**缺点**:
- 改动量大
- 需要数据迁移
- 并发处理复杂

**预计工时**: 12h

**推荐**: 方案 A（利用现有基础设施，快速验证）
