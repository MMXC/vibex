## 执行摘要

### 背景

DDD 建模（Context Map/Process Tree/Component Tree）产出了限界上下文、业务流程和组件结构，但缺少一个**可视化、可交互的原型层**。当前 DDD Canvas 与前端实现之间存在 Gap：PM 无法快速生成可演示的原型，前端工程师无法直接获得 Mock 数据和页面路由信息。

### 目标

在 Sprint 1 构建一个**可视化原型画布**，支持拖拽布局、Mock 数据绑定、页面路由列表和 JSON 导出，使 DDD 建模产出能够直接转化为可交互原型。

### 成功指标

| 指标 | 目标值 |
|------|--------|
| 组件拖入画布成功率 | ≥ 95%（无崩溃、无数据丢失） |
| Mock数据绑定保存率 | ≥ 98%（刷新页面后数据保留） |
| JSON 导出成功率 | 100%（含 Mock数据 + 路由树） |
| Round-trip 数据完整性 | Mock数据 + 路由树 100% 保留 |
| Sprint 1 按时交付 | ≤ 8h 工时 |

## 2a. 本质需求穿透

### Epic 1: 拖拽布局编辑器

**用户底层动机**: 不写代码就能"看到"页面长什么样。DDD 产出了组件树，但组件树是结构化的数据，不是可视化的页面。用户在"脑子里排列组件"和"实际看到布局"之间需要一个桥梁。

**去掉现有方案的理想解法**: 理想解法是自然语言描述页面，AI 直接生成布局。但这是 Phase 2 的目标。Phase 1 的理想解法是：给用户一个无限大的白板，上面有所有组件的卡片，用户像拼图一样把它们拼成页面。

**解决的本质问题**: 将**结构化的组件描述**转化为**可视化的页面布局**。消除"我描述的"和"你理解的"之间的歧义，让 PM 和前端在同一个视觉载体上对齐。

### Epic 2: Mock数据绑定

**用户底层动机**: 工程师看到空按钮、空列表，无法判断"这个按钮点击后会不会展示数据"。Mock 数据让组件"活"起来，看到实际的数据格式和渲染效果，减少"返工"。

**去掉现有方案的理想解法**: 理想解法是后端 API 已经就绪，组件直接连真实数据。但 Sprint 1 没有后端。Mock 数据是"在真实数据到来之前，让组件看起来像有真实数据"。

**解决的本质问题**: 将**组件静态展示**转化为**组件动态渲染**。消除"看起来有数据"和"真的有数据"之间的认知 Gap，让团队在 Sprint 0 就知道数据字段长什么样。

### Epic 3: 路由树

**用户底层动机**: 原型不只是单页面，是一个有导航的多页面系统。用户需要知道"这个原型有几个页面"、"页面之间怎么跳转"。路由树是这种导航关系的结构化表达。

**去掉现有方案的理想解法**: 理想解法是直接画出页面之间的连线（谁跳谁），像流程图一样。但这是 Phase 2 的 DAG 视图。Phase 1 的理想解法是：先让用户知道有哪些页面，页面之间怎么跳转。

**解决的本质问题**: 将**页面列表**转化为**导航关系图**（简化版）。消除"只知道有哪些页面"和"知道页面之间怎么组织"之间的认知 Gap。

### Epic 4: JSON导出增强

**用户底层动机**: 画布上的原型需要被其他工具使用（存档、导入其他系统、给后端看数据结构）。JSON 是通用的交换格式，导出的 JSON 质量直接影响原型数据的生命力。

**去掉现有方案的理想解法**: 理想解法是原型数据直接存在数据库，所有工具实时读取。但 Sprint 1 需要导出功能作为数据持久化的中间层。

**解决的本质问题**: 将**画布内数据**转化为**可传输的标准格式**。消除"数据只在画布里"和"数据可以到处流通"之间的限制。

### Epic 5: 组件库完善

**用户底层动机**: 用户拖了一个 Button 到画布上，结果渲染出来是个灰色的方块，什么交互都没有。这会让用户对工具失去信任。组件库必须"所见即所得"。

**去掉现有方案的理想解法**: 理想解法是每个组件都有完整的样式、状态、交互。但 Phase 1 的约束是时间有限，只需要保证基础渲染正确即可。

**解决的本质问题**: 将**组件占位符**转化为**可识别的真实组件**。消除"拖了一个组件但不知道是什么"和"一眼认出这是什么组件"之间的歧义。

## 2c. 用户情绪地图

### 页面 1: 原型编辑器主画布

**进入情绪**: 中性偏期待。用户带着 DDD 产出的组件树来到画布，希望"把它们拼成页面"。期望：拖进来就能看到效果。

**迷路引导**:
- 找不到组件面板 → 左上角固定"组件库"Tab，点击展开
- 不知道能拖什么 → 组件面板顶部显示"拖拽以下组件到画布"
- 拖入后不知道干什么 → 节点上有 tooltip 提示"双击编辑属性"

**出错兜底**:
- 网络异常 → 画布显示离线提示 Banner，数据暂存本地 localStorage
- 组件拖入失败 → Toast 提示"组件加载失败，请重试"，画布不崩溃
- 数据保存失败 → Toast 提示"保存失败，正在重试..."，3次重试后提示"保存超时，数据已暂存本地"

### 页面 2: 属性面板（Mock数据Tab）

**进入情绪**: 有目标感。用户刚拖入一个列表组件，想给它绑定数据。期望：有个地方能填 Mock 数据，填完能看到效果。

**迷路引导**:
- 找不到 Mock Tab → Tab 标签显示"Mock数据"带图标，默认选中
- 不知道填什么格式 → Tab 内显示 JSON 示例模板，可一键填充
- 填错了不知道 → 实时 JSON 校验，错误时 Tab 顶部显示红色提示

**出错兜底**:
- JSON 格式错误 → 红色边框 + "JSON 格式错误，请检查语法"
- 数据超长 → 截断显示 + "数据过长（>50KB），建议精简"
- 接口超时 → "数据加载超时，请检查网络或减少数据量"

### 页面 3: 路由列表抽屉

**进入情绪**: 想了解整体结构。用户想知道这个原型有几个页面，页面之间怎么组织。期望：看到一个清晰的页面列表。

**迷路引导**:
- 找不到路由入口 → 左侧导航栏有"路由树"图标，点击展开抽屉
- 不知道点击会发生什么 → 列表项悬停显示"点击跳转到画布"

**出错兜底**:
- 无页面数据 → 空状态：插图 + "还没有添加页面，请先在画布上创建"
- 路由冲突 → 黄色警告 + "路由 /users 与 /users/:id 可能冲突"
