## P0 — 必须修复

### P-001: Auth 页面风格不统一（UI 一致性 P0）

**问题描述**: Auth 页面（`/auth`）至今仍使用浅色背景 + 内联样式，与整体深色玻璃态 + 赛博青风格完全割裂。

| 对比项 | 参照页面（Dashboard） | Auth 现状 |
|--------|---------------------|----------|
| 背景 | 深色 `#0a0a0f` + 网格 + 发光球 | 浅灰 `#f8f9fa` |
| 卡片 | 玻璃态 `backdrop-filter: blur()` | 白色 + 简单阴影 |
| 按钮 | 渐变 + 霓虹发光 | 蓝色实心 `#0070f3` |
| 输入框 | 深色 + 发光边框 | 浅色 + 灰边框 |

**影响**: 这是用户进入产品的第一个门槛，风格割裂直接影响品牌信任度和技术专业感。该问题在 `style-gap-analysis.md`（2026-03-04）中已明确标记为 P0，但至今未修复。

**根本原因**: 历史上 Auth 页面作为独立模块开发，未遵循 DESIGN.md 规范。

**验收标准**:
- [ ] Auth 页面使用 `auth.module.css`（非内联样式）
- [ ] 背景系统与 Dashboard 一致（网格叠加 + 发光球）
- [ ] 登录卡片使用玻璃态效果
- [ ] 按钮使用 `--gradient-primary` + 霓虹发光
- [ ] 无硬编码颜色值，全部使用 CSS 变量

**建议方案**:
1. 创建 `auth.module.css`，复用 Dashboard 背景系统
2. 将内联样式迁移到 CSS Module
3. 应用设计变量 `--color-primary` / `--color-bg-glass` 等
4. 验收后更新 DESIGN.md 维护指南

### P-002: 需求智能补全缺失（生成质量 P0）

**问题描述**: 用户在需求输入框填写模糊描述时，系统直接调用 AI 生成，缺乏中间澄清环节。导致：输入模糊 → 直接生成 → 结果不符合预期 → 用户放弃。

**已知情况**: 此问题在 2026-04-11 的 PM 提案（`vibex-pm-proposals-20260411`）中已识别，并建议了方案 A（前端实时检测 + 后端澄清 API，4h），但至今未进入开发阶段。

**产品影响**: 这是影响"生成质量"核心指标的关键功能。缺少澄清机制意味着产品无法真正服务于"不知道如何描述 DDD 需求"的目标用户。

**验收标准**:
- [ ] 用户输入过程中，系统检测关键词（实体名、动词）
- [ ] 必要时弹出澄清追问卡片（如"你提到的'用户'是指注册会员还是访客？"）
- [ ] 用户回答后，需求描述自动补充
- [ ] 经澄清后的需求，AI 生成结果评分 +20%（需配合 P-010）
- [ ] 澄清响应时间 < 1s

**建议方案**（来自 2026-04-11 分析）:
- 前端监听输入事件，debounce 300ms 后检测关键词
- 后端提供轻量 `/api/v1/analyze/partial` 接口（或基于规则的澄清）
- 前端渲染澄清卡片，用户可跳过
- 预计工时：4h

### P-003: 项目搜索缺失（核心体验 P0）

**问题描述**: 用户在 Dashboard 拥有 10+ 项目后，无法快速定位目标项目，只能靠记忆或滚动查找。当项目数量达到 50+ 时，体验严重退化。

**已知情况**: 此问题同样在 2026-04-11 提案中识别，建议方案 A（前端过滤 + debounce，2h），未开发。

**产品影响**: 直接影响高频用户（多项目管理者）的使用效率，是企业场景的基础需求。

**验收标准**:
- [ ] Dashboard 显示搜索框，输入即过滤
- [ ] 支持按项目名称过滤
- [ ] 支持按创建时间排序（最新/最旧）
- [ ] 支持按状态过滤（活跃/归档）
- [ ] 搜索响应时间 < 200ms
- [ ] 无搜索结果时显示友好提示

**建议方案**:
- 前端使用现有 API 返回的全量项目列表（< 100 个时无性能问题）
- `filter()` + `debounce(300ms)` 实现客户端搜索
- 预计工时：2h

## P1 — 优先实现

### P-004: Canvas 页面 Phase 导航混乱（交互流程 P1）

**问题描述**: Canvas 页面包含多个 Phase（上下文、领域模型、流程、组件、原型），但 Phase 间的导航逻辑存在以下问题：

1. **TabBar 行为变更风险**: 2026-04-13 将 TabBar 从 `disabled` 改为无障碍化（所有 tab 可点击），但 prototype tab 的 phase 切换行为（`setPhase('prototype')` + `setActiveTree('component')`）与其他 tab 不对称，用户可能困惑。
2. **继续·流程树按钮**: 点击后 phase 切换，用户可能不清楚自己到了哪个阶段。
3. **Mobile prototype tab**: 新增的移动端 prototype tab 与桌面端行为是否一致未经充分验证。

**产品影响**: Canvas 是产品的核心操作区，Phase 导航混乱直接影响"能顺利走完流程"的产品哲学。

**验收标准**:
- [ ] TabBar 所有 tab 点击行为一致（setPhase + setActiveTree 对称）
- [ ] Phase 切换时，页面顶部或工具栏显示当前阶段名称（进度指示）
- [ ] Phase 切换有平滑过渡动画（200ms ease-out）
- [ ] "继续·流程树"点击后，视觉反馈明确（按钮 loading + 页面状态变化）
- [ ] E2E 测试覆盖所有 Tab 点击 → Phase 切换路径

### P-005: 团队协作 UI 完全缺失（企业场景 P1）

**问题描述**: KV 后端（`CollaborationService`）已部署，支持 Owner/Member/Viewer 三级权限。但前端团队协作 UI 为零：
- 无法创建团队
- 无法邀请成员
- 无法管理权限
- 无法看到在线协作者

**产品影响**: 团队协作是企业场景的硬需求，KV 后端空转意味着"企业版 VibeX"无法真正落地。

**验收标准**:
- [ ] 用户可创建团队（团队名称 + 描述）
- [ ] 团队 Owner 可邀请成员（输入邮箱 → 发送邀请）
- [ ] 成员列表显示角色（Owner/Member/Viewer）
- [ ] Owner 可修改成员角色或移除成员
- [ ] 多人同时编辑时，实时显示在线成员头像
- [ ] 权限冲突时有友好提示（"张三分组正在修改，请稍后"）

**建议方案**（来自 2026-04-11 分析）:
- 复用已有 KV 后端，新增 `/api/v1/teams` 系列接口
- 前端新增团队设置页面 + 协作视图
- 预计工时：6h

### P-006: 版本历史 projectId 刚修复，对比功能空白（可追溯性 P1）

**问题描述**: 
1. **2026-04-14 最新修复**: Canvas 版本历史 projectId 链路刚刚修复（`dd482541`, `438af56f`），说明该功能基础尚不稳固。
2. **版本对比缺失**: 即使查看历史版本，用户也无法对比两个版本的差异（需求变更了什么？模型改了哪些？）。

**产品影响**: 版本对比是企业场景的核心需求，也是个人用户迭代优化的基础。当前只能"看"历史，无法"对比"历史。

**验收标准**:
- [ ] 版本列表正常加载（projectId 链路稳定）
- [ ] 点击任意版本，可预览该版本快照
- [ ] 支持选择两个版本进行 diff 对比（高亮差异字段）
- [ ] 对比视图显示：新增内容（绿色）、删除内容（红色）、修改内容（黄色）
- [ ] 支持将旧版本恢复到当前（需二次确认）

### P-007: 导入/导出功能缺失（效率 P1）

**问题描述**: 用户无法将需求文档（.md/.json/.yaml）导入系统，也无法将生成结果导出为可复用格式。导致：
- 已有 PRD 文档的用户需要手动输入，效率极低
- 生成结果无法沉淀为可维护的文档资产

**验收标准**:
- [ ] 支持导入 `.md` / `.json` / `.yaml` 格式的需求文档
- [ ] 导入时解析结构化字段（bounded context / domain model / business flow）
- [ ] 支持导出需求文档（Markdown 格式）
- [ ] 支持导出设计产物（Mermaid 图代码、JSON schema）
- [ ] 导出文件可下载，文件名包含项目名和时间戳

**建议方案**:
- 导入：新增 `/api/v1/projects/import` 接口，支持文件上传 + 结构化解析
- 导出：前端提供下载功能，调用现有 API 拼接格式
- 预计工时：2h

## P2 — 锦上添花

### P-008: Canvas Tree 按钮样式不一致（UI 一致性 P2）

**问题描述**: Canvas 页面的三个树（BoundedContextTree / BusinessFlowTree / ComponentTree）的工具栏按钮样式不统一。按钮图标、文字、hover 效果存在细微差异，影响视觉一致性。

**已知情况**: 此问题在 2026-04-11 提案中识别，建议了 Tree 按钮统一方案（P005）。

**验收标准**:
- [ ] 三个树工具栏按钮样式 100% 一致（图标 + 文字 + 间距 + hover）
- [ ] 截图回归测试覆盖所有树工具栏
- [ ] 与 DESIGN.md 按钮规范一致（Secondary 按钮样式）

---

### P-009: 快捷键系统缺失（效率用户 P2）

**问题描述**: 效率型用户（高频使用）缺乏键盘快捷键支持，每次操作都需要鼠标，降低操作速度。

**已知情况**: 2026-04-11 提案已计划，工时 1h，但未开发。

**验收标准**:
- [ ] `Ctrl+S`: 保存当前输入草稿
- [ ] `Ctrl+Enter`: 提交需求/确认下一步
- [ ] `Ctrl+Z`: 撤销（confirmationStore 历史记录）
- [ ] `?`: 打开快捷键帮助面板
- [ ] 帮助面板列出所有可用快捷键

---

### P-010: 错误提示样式不统一（UI 一致性 P2）

**问题描述**: 不同页面、不同场景的错误提示（toast / inline / modal）样式不统一：
- Canvas 页面的 error toast 样式与 Dashboard 不同
- API 错误消息有时是纯技术描述（如 "401 Unauthorized"），用户无法理解
- 401 错误在 2026-04-13 刚修复（`vibex-auth-401-redirect`），但其他 HTTP 错误码未统一处理

**验收标准**:
- [ ] 全站使用统一的 `useToast` 组件
- [ ] 错误消息映射表：将 HTTP 状态码转为用户友好文字（如 500 → "服务器开小差了，请稍后重试"）
- [ ] Toast 支持重试按钮（适用于可恢复错误）
- [ ] Inline 错误提示与 input 样式一致（红色边框 + 错误文字）

## P3 — 长期优化

### P-011: 页面状态反馈不清晰（体验 P3）

**问题描述**: Canvas 页面 AI 生成过程中，加载状态不够明确：
- "生成中" 仅靠 spinner，不清楚进度或阶段
- 生成失败后，用户的挽救路径不明确

**验收标准**:
- [ ] AI 生成时显示进度阶段（"正在分析需求..." → "正在生成上下文..." → "完成"）
- [ ] 生成失败时显示原因 + 可操作按钮（如"重新生成"/"修改需求后重试"）
- [ ] 使用 DESIGN.md 动效规范（ease-spring，loading 时按钮 disable + loading spinner）

---

### P-012: AI 生成质量反馈机制缺失（数据闭环 P3）

**问题描述**: 用户无法对 AI 生成结果评分（1-5 星），产品无法收集数据来持续优化生成质量。

**验收标准**:
- [ ] 生成结果页面显示评分入口（1-5 星 + 可选文字反馈）
- [ ] 评分提交后持久化到后端
- [ ] 评分数据可用于后续 AI 调优
- [ ] 用户可查看自己的历史评分记录
