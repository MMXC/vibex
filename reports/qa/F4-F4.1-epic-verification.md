## 4. 🔴 Critical Bug Found During Testing

### Bug: F4.2 未实施导致 F4.1 引入新问题

**位置**: `BoundedContextTree.tsx:458-461`

```typescript
const handleConfirmAll = useCallback(() => {
  // Advance phase (no confirm gating in Epic 3)
  advancePhase();
}, [contextNodes, advancePhase]);
```

**问题**: `handleConfirmAll` 没有设置任何节点的 `status === 'confirmed'`。配合 F4.1（`allConfirmed` 改为检查 `status`），后果是：

1. 用户勾选所有节点的 checkbox → `status === 'confirmed'`（通过 `toggleContextNode`）
2. `allConfirmed === true` ✅
3. 用户点击"确认所有 → 继续到流程树"
4. **`handleConfirmAll` 不设置 `status`，只调用 `advancePhase()`**
5. Phase 推进到 Flow Tree，但 Context Tree 节点 `status` 未变
6. **用户返回 Context Tree 时，所有节点 `status` 仍为 `pending`，`allConfirmed === false`，按钮重新显示"确认所有"**

**正确行为**: `handleConfirmAll` 应先将所有节点 `status` 设为 `'confirmed'`，再调用 `advancePhase()`。

**影响**: 核心用户路径被破坏——点击"确认所有"后，Context Tree 状态未真正固化，用户再次访问时需重新确认。

**Spec 明确说明**:
> "此修复依赖 F4.2（`handleConfirmAll` 必须真正设置 `status === 'confirmed'`）。若 F4.2 未实施，`allConfirmed` 永远为 `false`，面板永远锁定。"

**严重程度**: High（破坏确认→继续流程的核心交互路径）

**建议**: F4.2（`handleConfirmAll` 原子性设置 `status`）必须与 F4.1 配对实施。建议 tester 任务 `tester-f4-f4.1` 标记为 `blocked`，等待 F4.2 完成后再验收。
